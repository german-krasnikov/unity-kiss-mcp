# T1 — AssetDatabase.Refresh / ImportAsset semantics

## (a) Refresh synchronicity vs compilation / domain reload

- [U6000.0–U6000.3 | HIGH | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/AssetDatabase.Refresh.html] Unity 6 docs state verbatim: "This operation happens synchronously for asset imports, and asynchronously for script compilation" — i.e. `Refresh()` returns after asset imports complete, but BEFORE compilation/domain reload finish.
- [U2021.3–U2022.3 | HIGH | https://docs.unity3d.com/2021.3/Documentation/ScriptReference/AssetDatabase.Refresh.html] The sync/async sentence is ABSENT from 2021.3 and 2022.3 doc pages (verified both); it's a Unity-6-era doc clarification, not a behavior change claim.
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.0/Documentation/Manual/AssetDatabaseRefreshing.html] Refresh process order: 1) scan Assets+Packages for adds/mods/deletes → 2) "Import and compile code-related files such as .dll, .asmdef, .asmref, .rsp, and .cs files" → 3) "Reload the scripting domain, if Refresh was not invoked from a script" → 4) post-process code assets → 5) import non-code assets → 6) hot-reload. Same steps in 2021.3 manual (https://docs.unity3d.com/2021.3/Documentation/Manual/AssetDatabaseRefreshing.html).
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/2021.3/Documentation/Manual/AssetDatabaseRefreshing.html] Consequence of step 3: when `Refresh()` is called FROM SCRIPT, the domain reload is never performed inside the call — it happens after your C# code finishes. Your code never observes the post-reload domain in the same call stack.
- [U2022.3–U2022.3 | HIGH | https://docs.unity3d.com/2022.3/Documentation/ScriptReference/AssetDatabase.ScheduleRefresh.html] `AssetDatabase.ScheduleRefresh` docs confirm the mechanics: direct `Refresh()` with scripts+dependent assets changed together imports dependent assets TWICE (once before domain reload, again after); ScheduleRefresh defers to "the next Editor application tick" so "the refresh does not occur while your C# code is running, therefore the domain reload occurs first".
- [U2022.3–U2022.3 | MED | https://docs.unity3d.com/2022.3/Documentation/ScriptReference/AssetDatabase.ScheduleRefresh.html] `ScheduleRefresh` is documented ONLY in 2022.x: page 404s in 2021.3, 2023.2, and 6000.0 docs (all three verified) — treat as 2022-only API surface.
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/AssetDatabase.Refresh.html] "This method implicitly triggers an asset garbage collection" (Resources.UnloadUnusedAssets) — present in 2021.3 and 6000.0 pages.
- NOT FOUND: official statement on what happens if `Refresh()` is called while another refresh is already in progress (queue vs merge).

## (b) ImportAssetOptions flags (names identical 2021.3 → 6000.3, verified on both enum pages)

- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.3/Documentation/ScriptReference/ImportAssetOptions.html] Exactly 6 members, no obsolete ones: `Default`, `ForceUpdate`, `ForceSynchronousImport`, `ImportRecursive`, `DontDownloadFromCacheServer`, `ForceUncompressedImport`.
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/ImportAssetOptions.html] `Default` — "Default import options."
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/ImportAssetOptions.ForceUpdate.html] `ForceUpdate` — "User initiated asset import. This should be passed when asset import is caused by the user (e.g. by clicking Reimport). Non-user initiated import happens when the asset modification time changes." → the flag to force reimport of a .cs whose mtime/content hash didn't change.
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/ImportAssetOptions.ForceSynchronousImport.html] `ForceSynchronousImport` — "Import all assets synchronously. By default some assets can be imported asynchronously (e.g. scripts can be compiled in the background). … when importing a scripts + prefabs, scripts have to be fully compiled before Prefab is serialized, otherwise it might get old variables." → it orders compilation before dependent-asset serialization WITHIN the refresh; per (a) step 3 it still cannot domain-reload inside your running script.
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/ImportAssetOptions.html] `ImportRecursive` — "When a folder is imported, import all its contents as well." (only meaningful when path is a folder; irrelevant for single .cs).
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/ImportAssetOptions.html] `DontDownloadFromCacheServer` — exact name confirmed — "Force a full reimport but don't download the assets from the cache server."
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/ImportAssetOptions.html] `ForceUncompressedImport` — "Forces asset import as uncompressed for edition facilities." (texture-oriented; irrelevant for .cs).
- [U2019–U6000.x | LOW | https://discussions.unity.com/t/assetdatabase-importasset-with-importassetoptions-forcesynchronousimport-not-synchronous/680321] Community-reported (no staff reply): `ImportAsset(..., ForceSynchronousImport)` called from INSIDE a ScriptedImporter does not run synchronously — the import starts only after the importer method returns; works as expected outside ScriptedImporter context.

## (c) ImportAsset(path) vs Refresh() for recompilation

- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/AssetDatabase.ImportAsset.html] `ImportAsset` imports only the asset at the given project-relative path (doc example explicitly shows sibling files are NOT imported); triggers AssetModificationProcessor.OnWillSaveAssets and AssetPostprocessor.OnPostprocessAllAssets.
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.0/Documentation/Manual/AssetDatabaseRefreshing.html] `ImportAsset` is one of the "AssetDatabase API[s] that queues an additional refresh, such as MoveAsset, CreateAsset and ImportAsset" — and that refresh pipeline "imports and compiles code-related files such as … .cs files"; so targeted `ImportAsset("Assets/Foo.cs")` is sufficient to trigger recompilation of changed scripts, no full `Refresh()` needed.
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/AssetDatabase.Refresh.html] `Refresh()` is the broader scan: it imports everything "that ha[s] modified or added content, import settings, or dependencies since the last refresh" and removes deleted assets — use it when you don't know the changed paths; use `ImportAsset` when you do.

## (d) Main thread + StartAssetEditing scope

- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.3/Documentation/Manual/programming-best-practices.html] "the core runtime is single-threaded and most APIs in the UnityEngine and UnityEditor namespaces can only be called from the main thread" — AssetDatabase included; no per-method doc note exists on the Refresh page itself.
- [U3.x–U6000.x | LOW | https://answers.unity.com/questions/52013/assetdatabaserefresh-of-thread.html] Community confirmation: calling `AssetDatabase.Refresh()` from a worker thread throws; standard workaround is flag + call from main-thread update loop.
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/AssetDatabase.StartAssetEditing.html] Inside Start/StopAssetEditing the AssetDatabase "Pauses automatic asset import"; all queued changes are processed as ONE import at the balancing `StopAssetEditing`; calls nest and counts must be equal; "some AssetDatabase APIs won't work as expected" during the pause because assets "aren't fully created in the asset database before the call to StopAssetEditing".
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/2021.3/Documentation/Manual/AssetDatabaseBatching.html] During the pause the database "does not process any further changes to your Assets"; if `StopAssetEditing` is never called "the Editor appears to be unresponsive when it comes to any Asset-related operations (Importing, Refreshing, etc.), and requires an Editor restart" — i.e. `Refresh()` inside the scope is deferred/inert until the counter balances, never an immediate import.
- NOT FOUND: doc sentence specifically naming `Refresh()`'s exact behavior (silent defer vs warning) when invoked inside a Start/StopAssetEditing scope — only the pause/unresponsive wording above.

## (e) No-op vs directory monitoring; Packages/ coverage

- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/AssetDatabase.Refresh.html] "Most of the time, you don't need to call this method yourself, because the Editor automatically detects changes to assets on disk and imports them as needed"; Refresh imports only what changed "since the last refresh" — so after the Editor already auto-imported a change, a manual `Refresh(Default)` finds nothing new to import (it still performs the disk scan). The "effectively no-op" reading is inference from this wording — flagged, no doc says "no-op" literally.
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.0/Documentation/Manual/AssetDatabaseRefreshing.html] Refresh scans BOTH folders: "it scans the Assets and Packages folders in your Project to check if any files have been added, modified, or deleted since the last scan" — so Packages/ (including local `file:` and embedded packages) IS covered, not only Assets/.
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.0/Documentation/Manual/upm-concepts.html] Only Local and Embedded package sources are mutable ("You can permanently change content only from Local and Embedded package sources"); registry/built-in/tarball packages are immutable — edits there aren't a supported change source for Refresh pickup.
- NOT FOUND: official statement on whether ForceUpdate-style reimport behaves differently for files under immutable (registry) packages.

## Key takeaways for the canonical reference

1. `Refresh()` = sync for asset imports, async for compilation (Unity 6 docs verbatim); domain reload NEVER happens inside a script-invoked Refresh (Manual step 3, all versions).
2. All 6 `ImportAssetOptions` names are stable 2021.3→6000.3; `DontDownloadFromCacheServer` confirmed exact; for .cs the only practically relevant flags are `ForceUpdate` (force reimport of unchanged file) and `ForceSynchronousImport` (compile-before-dependent-serialize ordering, not inline reload).
3. Targeted `ImportAsset` on a .cs queues a refresh whose pipeline compiles code files — full `Refresh()` not required for recompile.
4. Main-thread only (UnityEditor namespace rule); inside Start/StopAssetEditing everything including Refresh is deferred until the nest counter balances.
5. Packages/ is scanned alongside Assets/; only local/embedded packages are mutable sources of change.
