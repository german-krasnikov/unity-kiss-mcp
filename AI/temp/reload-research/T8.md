# T8 — Reload/Recompile Survival in Open-Source Unity MCP Bridges

## 1. CoplayDev/unity-mcp (ex justinpbarnett/unity-mcp; Python server + C# plugin, stdio/http transports)

- [CoplayDev/unity-mcp@main | HIGH | https://github.com/CoplayDev/unity-mcp/blob/main/MCPForUnity/Editor/Tools/RefreshUnity.cs] (a) Dedicated `RefreshUnity` tool: `AssetDatabase.Refresh(ImportAssetOptions.ForceUpdate | ForceSynchronousImport)` (lightweight variant = `ForceSynchronousImport` only); `compile:"request"` → `CompilationPipeline.RequestScriptCompilation()`.
- [CoplayDev/unity-mcp@main | HIGH | https://github.com/CoplayDev/unity-mcp/blob/main/MCPForUnity/Editor/Tools/RefreshUnity.cs] (a/d) `wait_for_ready` polls via `EditorApplication.update` tick until `!isCompiling && !isUpdating && !TestRunStatus.IsRunning` (60s cap); when compile was requested on Unity 6+, it returns immediately with `resulting_state` ("compiling"/"asset_import"/"idle") and tells the client to poll — explicit "respond before reload, poll after" contract.
- [CoplayDev/unity-mcp@main | HIGH | https://github.com/CoplayDev/unity-mcp/blob/main/MCPForUnity/Editor/Services/StdioBridgeReloadHandler.cs] (b) `AssemblyReloadEvents.beforeAssemblyReload` → `TransportManager.StopAsync(Stdio)` with `stopTask.Wait(500)`, sets `EditorPrefs` flag `ResumeStdioAfterReload`, writes `StdioBridgeHost.WriteHeartbeat(true, "reloading")`; `afterAssemblyReload` → `ResumeStdioWithRetriesAsync()` with 6-attempt escalating schedule (0s→1s→3s…30s), `WriteHeartbeat(false,"stopped")` on final failure.
- [CoplayDev/unity-mcp@main | HIGH | https://github.com/CoplayDev/unity-mcp/blob/main/Server/src/transport/plugin_hub.py] (b) Python hub treats reload as connection-loss: waits up to 20s for a plugin session to reappear (`UNITY_MCP_SESSION_RESOLVE_MAX_WAIT_S`, 50–250ms poll); in-flight futures get `PluginDisconnectedError` → response `hint="retry"`; ping-probe readiness gate (~6s) before fast-path commands; stale-socket detection via ping/pong (20s timeout).
- [CoplayDev/unity-mcp@main | HIGH | https://github.com/CoplayDev/unity-mcp/blob/main/Server/src/transport/unity_transport.py] (b) `retry_on_reload=True` kwarg threaded into `PluginHub.send_command_for_instance`; timeouts normalized into explicit `hint='retry'` responses instead of opaque `asyncio.TimeoutError`.
- [CoplayDev/unity-mcp@main | HIGH | https://github.com/CoplayDev/unity-mcp/blob/main/README.md] (c) Installed as remote UPM git package (`https://github.com/CoplayDev/unity-mcp.git?path=/MCPForUnity#main`); immutable PackageCache → no `file:` self-update problem (they never edit their own package from the tool).
- [CoplayDev/unity-mcp@main | HIGH | https://github.com/CoplayDev/unity-mcp/issues/1173] (e) TcpListener leak after domain reload regressed in 9.7.1/Windows: original fix (PR #688 for #692) = `ExclusiveAddressUse = true`, `listener?.Server?.Dispose()` in beforeAssemblyReload cleanup, 2000ms listener-task wait; regression cut wait to `toWait.Wait(500)` (too short for Windows socket release) and `PortManager.DiscoverNewPort()` silently rebinds to a new port while Python stays on the old one → hangs.
- [CoplayDev/unity-mcp@main | MED | https://github.com/CoplayDev/unity-mcp/issues/672] (e) Open DX request for auto-connect/reconnect & keep-server-running; v9.4.6/9.4.7 release line added stale-connection detection + "domain reload resilience for script tools".

## 2. CoderGamester/mcp-unity (Node/TS MCP server = WebSocket CLIENT; WS server lives inside Unity)

- [CoderGamester/mcp-unity@main | HIGH | https://github.com/CoderGamester/mcp-unity/blob/main/Editor/Tools/RecompileScriptsTool.cs] (a) `recompile_scripts` tool: `if (!EditorApplication.isCompiling) CompilationPipeline.RequestScriptCompilation();` then subscribes `CompilationPipeline.assemblyCompilationFinished` + `compilationFinished` and completes queued `TaskCompletionSource`s with results (incl. compile logs) inside `OnCompilationFinished` — i.e. response is flushed after compile but **before** the domain reload kills the socket.
- [CoderGamester/mcp-unity@main | HIGH | https://github.com/CoderGamester/mcp-unity/blob/main/Editor/UnityBridge/McpUnityServer.cs] (b/d) `[InitializeOnLoad]` singleton; `OnBeforeAssemblyReload → StopServer()`; `OnAfterAssemblyReload` → `ScheduleStartServer(reason:"assembly reload")` which uses `EditorApplication.delayCall` + `EditorApplication.timeSinceStartup` earliest-time gating (deliberate delayed rebind, not immediate); no SessionState — restart decision driven by persisted `AutoStartServer` setting.
- [CoderGamester/mcp-unity@main | HIGH | https://github.com/CoderGamester/mcp-unity/blob/main/Server~/src/unity/unityConnection.ts] (b) Node-side reconnect: exponential backoff `baseDelay * multiplier^attempt` (1s→30s) + 0–20% jitter, max 50 attempts (fd-leak guard, issue #110); Play-mode close code `4001` switches to flat 3s polling with unlimited retries; WS ping every 30s, stale if no pong in 5s; `forceReconnect()` resets attempts.
- [CoderGamester/mcp-unity@main | HIGH | https://github.com/CoderGamester/mcp-unity/blob/main/Server~/src/unity/mcpUnity.ts] (b) In-flight handling: on drop, `rejectAllPendingRequests(CONNECTION error)`; with `queueIfDisconnected`, commands are queued during reconnect and replayed ("Replaying ${commands.length} queued commands"); default request timeout 10s, per-request override.
- [CoderGamester/mcp-unity@main | HIGH | https://github.com/CoderGamester/mcp-unity/blob/main/README.md] (c/e) Install = UPM git URL; README documents the failure mode verbatim: "bridge connection is lost when the domain reloads upon switching to Play Mode. The workaround is to turn off Reload Domain in … Enter Play Mode Settings" — they punt Play-mode reload to user config.

## 3. Arodoid/UnityMCP (early/simple; Node WS SERVER :8080, Unity is the WS client)

- [Arodoid/UnityMCP@main | HIGH | https://github.com/Arodoid/UnityMCP/blob/main/UnityMCPPlugin/Editor/UnityMCPConnection.cs] (a) No refresh/recompile trigger at all — no `AssetDatabase.Refresh`, no `CompilationPipeline` calls; only `AssetDatabase.FindAssets` for scanning.
- [Arodoid/UnityMCP@main | HIGH | https://github.com/Arodoid/UnityMCP/blob/main/UnityMCPPlugin/Editor/UnityMCPConnection.cs] (b) Reload survival = accidental: `[InitializeOnLoad]` static ctor re-runs after reload and a dumb 5s retry loop (`reconnectInterval = 5f`, 5s connect timeout, manual `RetryConnection()`); no `AssemblyReloadEvents`, no `SessionState`.
- [Arodoid/UnityMCP@main | HIGH | https://github.com/Arodoid/UnityMCP/blob/main/unity-mcp-server/src/index.ts] (b) Server side: on disconnect just `unityConnection = null`, new requests throw "Unity Editor is not connected"; no queue/replay; in-flight command promises can hang past the 5s timeout window.
- [Arodoid/UnityMCP@main | HIGH | https://github.com/Arodoid/UnityMCP/blob/main/README.md] (c) Install = "Copy the `UnityMCPPlugin` folder to your Unity project's Assets directory" — lives in Assets/, so plugin-source edits reload like normal project scripts.
- NOT FOUND: dedicated issue/PR about connection-lost-after-compile in Arodoid/UnityMCP (repo effectively dormant).

## 4. IvanMurzak/Unity-MCP (SignalR client in Unity → .NET server; connection core shipped as McpPlugin NuGet DLLs)

- [IvanMurzak/Unity-MCP@main | HIGH | https://github.com/IvanMurzak/Unity-MCP/blob/main/Unity-MCP-Plugin/Packages/com.ivanmurzak.unity.mcp/Editor/Scripts/API/Tool/Assets.Refresh.cs] (a) `Assets.Refresh` tool: `AssetDatabase.Refresh(options ?? ImportAssetOptions.ForceSynchronousImport)` on a main-thread runner; if `EditorApplication.isCompiling` afterwards it returns `ResponseCallTool.Processing(...).SetRequestID(requestId)` immediately and schedules a post-compilation follow-up message — a two-phase async response over SignalR (the in-flight-request answer).
- [IvanMurzak/Unity-MCP@main | HIGH | https://github.com/IvanMurzak/Unity-MCP/blob/main/Unity-MCP-Plugin/Packages/com.ivanmurzak.unity.mcp/Editor/Scripts/Startup.Editor.cs] (b) `beforeAssemblyReload` → `TryDisconnectAndCleanup(onlyIfConnected: true)` (deliberately skips disconnect if a connect attempt is mid-flight to avoid hanging); `afterAssemblyReload` → rebuild plugin + reconnect gated on `KeepConnected` / CI checks; `OnPlayModeStateChanged`: dispose on ExitingPlayMode, delayed + immediate reconnection on EnteredEditMode; no SessionState — relies on re-init after reload.
- [IvanMurzak/Unity-MCP@main | HIGH | https://github.com/IvanMurzak/Unity-MCP/blob/main/README.md] (c) Install: OpenUPM (`openupm add com.ivanmurzak.unity.mcp`), installer .unitypackage, or `npx unity-mcp-cli install-plugin`; in their own dev repo the plugin sits as an **embedded package** at `Unity-MCP-Plugin/Packages/com.ivanmurzak.unity.mcp` — embedded = mutable + normal reload, which is how they dogfood self-edits.
- [IvanMurzak/Unity-MCP@main | MED | https://github.com/IvanMurzak/Unity-MCP/wiki/Troubleshooting] (e) Official stance: "disconnects before a domain reload and reconnects afterward… brief disconnections are normal"; recommends Enter Play Mode Settings = no domain reload for uninterrupted runs.

## 5. hatayama/unity-cli-loop (ex uLoopMCP; TS server polls a TCP server inside Unity) — best-in-class reload design

- [hatayama/unity-cli-loop@main | HIGH | https://github.com/hatayama/unity-cli-loop/blob/main/Packages/src/Editor/Server/McpServerController.cs] (b) `beforeAssemblyReload` → `DomainReloadRecoveryUseCase.ExecuteBeforeDomainReload()`; `afterAssemblyReload` → `ExecuteAfterDomainReloadAsync()`; restore reads `McpEditorSettings.GetCustomPort()` + `GetIsAfterCompile()`, rebinds saved port with fallback, `SaveRunningServerSession(port, projectRoot, sessionId)`.
- [hatayama/unity-cli-loop@main | HIGH | https://github.com/hatayama/unity-cli-loop/blob/main/Packages/src/Editor/Api/McpTools/UseCases/System/DomainReloadRecoveryUseCase.cs] (b) Before reload: correlation-ID'd `DomainReloadDetectionService.StartDomainReload(...)`, graceful server stop + client notifications, **throws if shutdown fails to prevent port conflicts**; after reload: `SessionRecoveryService.RestoreServerStateIfNeeded()`, pending-compile-request processing (currently behind a disabled feature flag), then `SendToolNotificationAfterCompilationAsync()` (MCP tools-changed handshake).
- [hatayama/unity-cli-loop@main | HIGH | https://github.com/hatayama/unity-cli-loop/blob/main/Packages/src/Editor/Api/McpTools/UseCases/Tools/CompileUseCase.cs] (b/d) Compile responses survive reload by persistence: `CompileResultPersistenceService.SaveResult(parameters.RequestId, response)` with IDs `compile_{unixTimeMs}_{correlationId}` when `WaitForDomainReload` — result is stored and delivered after the reload instead of racing the socket.
- [hatayama/unity-cli-loop@main | HIGH | https://github.com/hatayama/unity-cli-loop/blob/main/Packages/src/Editor/Core/ApplicationServices/CompilationLockService.cs] (d) Out-of-band compile signal: creates a lock FILE on `CompilationPipeline.compilationStarted` and deletes it on `compilationFinished` so external CLI/server can detect "Unity is compiling" without a live socket.
- [hatayama/unity-cli-loop@main | HIGH | https://github.com/hatayama/unity-cli-loop/blob/main/Packages/src/TypeScriptServer~/src/unity-discovery.ts] (b) TS side: singleton `UnityDiscovery` poller — fast `INITIAL_INTERVAL_MS` for first attempts then `EXTENDED_INTERVAL_MS`; single configured port (`UNITY_TCP_PORT`); `handleConnectionLost()` waits 2s then restarts discovery (gives Unity time to reload).
- [hatayama/unity-cli-loop@main | MED | https://github.com/hatayama/unity-cli-loop/blob/main/Packages/src/Editor/Core/ApplicationServices/CompilationExecutionService.cs] (a) Compile trigger is abstracted behind `CompileController.TryCompileAsync(forceRecompile, ct)`; README states refresh-then-compile (`AssetDatabase.Refresh()` first, incremental vs forced full recompile option) — exact `RequestScriptCompilation` call site not directly verified.
- [hatayama/unity-cli-loop@main | MED | https://github.com/hatayama/unity-cli-loop/blob/main/Packages/src/README.md] (c) Distributed via OpenUPM (immutable `Library/PackageCache` with hashed dir); in the dev repo the package is a local package at `Packages/src` of its own Unity project.

## Cross-repo findings

- NOT FOUND: any use of `EditorApplication.LockReloadAssemblies` in any of the five repos (uLoopMCP's "CompilationLockService" is a lock *file*, not assembly locking).
- [AnkleBreaker-Studio/unity-mcp-server@main | LOW | https://github.com/AnkleBreaker-Studio/unity-mcp-server] Sixth candidate (268-tool server) located but not investigated in depth this pass.
- Universal pattern: nobody keeps the socket alive through reload; everyone does stop-before/restart-after + external-side retry. The differentiators are (1) whether the in-flight request gets a real answer (CoderGamester: respond-inside-compilationFinished; IvanMurzak: `Processing` + follow-up push; uLoopMCP: persist result by requestId; CoplayDev: `hint="retry"` punt) and (2) whether reload state is persisted (EditorPrefs flag / session port file) vs re-derived from `[InitializeOnLoad]`.
