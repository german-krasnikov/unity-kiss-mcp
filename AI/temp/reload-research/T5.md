# T5 — Auto Refresh prefs & focus-based refresh (deliverable)

## (a) EditorPrefs keys, enum values, UI path

- [U4–U2021.2 | HIGH | https://discussions.unity.com/t/editorprefs-pre-defined-magic-strings/632359] Legacy key is boolean `kAutoRefresh` (`EditorPrefs.SetBool/SetInt("kAutoRefresh", 0|1)`), the long-standing "magic string" controlling editor auto-refresh.
- [U2021.2–U2021.2 | HIGH | https://raw.githubusercontent.com/Unity-Technologies/UnityCsReference/2021.2/Editor/Mono/PreferencesWindow/AssetPipelinePreferences.cs] The 2021.2 branch still reads only `EditorPrefs.GetBool("kAutoRefresh")` — `kAutoRefreshMode` does NOT exist in 2021.2; the "2021.2+" assumption is wrong.
- [U2021.3–U6000.x | HIGH | https://raw.githubusercontent.com/Unity-Technologies/UnityCsReference/2021.3/Editor/Mono/PreferencesWindow/AssetPipelinePreferences.cs] 2021.3 LTS (current patch line) uses int `kAutoRefreshMode` with `internal enum AssetPipelineAutoRefreshMode { Disabled = 0, Enabled = 1, EnabledOutsidePlaymode = 2 }`; legacy fallback: `GetInt("kAutoRefreshMode", GetBool("kAutoRefresh") ? Enabled : Disabled)`.
- [U2021.3–U2023.1 | MED | https://issuetracker.unity3d.com/issues/recompile-after-finish-playmode-option-is-gone] The mode-dropdown rework landed mid-LTS: the accompanying "Recompile After Finished Playing" removal appeared in 2021.3.10f1 / 2022.2.0b8 / 2023.1.0a12 (later partially reverted, case-1325047) — so `kAutoRefreshMode` arrived in 2021.3 around .10f1 as a backport, not at 2021.3.0.
- [U2022.1–U6000.x | HIGH | https://raw.githubusercontent.com/Unity-Technologies/UnityCsReference/2022.1/Editor/Mono/PreferencesWindow/AssetPipelinePreferences.cs] 2022.1+ branches contain the same `kAutoRefreshMode` + 3-value enum; identical in `master` (Unity 6 line) — Unity 6 has no new key, `kAutoRefreshMode` IS the Unity 6 equivalent.
- [U6000.0–U6000.x | HIGH | https://docs.unity3d.com/6000.0/Documentation/Manual/preferences-asset-pipeline.html] Unity 6 UI path: Edit > Preferences > Asset Pipeline (macOS: Unity > Settings > Asset Pipeline); Auto Refresh dropdown values exactly: "Disabled", "Enabled", "Enabled Outside Playmode".
- [U2021.3–U2021.3 | MED | https://docs.unity3d.com/2021.3/Documentation/Manual/Preferences.html] Docs lag warning: 2021.3 manual still documents Auto Refresh as a checkbox ("Enable this setting to import Assets automatically as they change") even though 2021.3 LTS code ships the 3-mode dropdown.

## (b) Directory Monitoring

- [U2021.3–U6000.x | HIGH | https://docs.unity3d.com/6000.0/Documentation/Manual/preferences-asset-pipeline.html] Directory Monitoring = "monitors directories instead of scanning all project files to detect asset changes" — a change-DETECTION optimization, not a background importer; documented as Windows-only in both 2021.3 and 6000.0 manuals.
- [U2021.3–U6000.x | HIGH | https://raw.githubusercontent.com/Unity-Technologies/UnityCsReference/master/Editor/Mono/PreferencesWindow/AssetPipelinePreferences.cs] Pref key is `EditorPrefs.GetBool("DirectoryMonitoring", true)` (default ON); UI is hard-disabled off-Windows via `Application.platform == RuntimePlatform.WindowsEditor` + help text "Directory monitoring currently only available on windows" — on macOS it is inert.
- NOT FOUND: any official doc describing Directory Monitoring × Auto Refresh interaction beyond "cheaper detection feeding the same refresh triggers" (no docs/changelog states it imports without focus).

## (c) Focus requirement / background refresh

- [U2019.3–U6000.x | HIGH | https://docs.unity3d.com/6000.0/Documentation/Manual/AssetDatabaseRefreshing.html] Canonical trigger list unchanged through Unity 6: refresh runs (1) "when the Unity Editor regains focus (if you have enabled Auto-Refresh in the Preferences window)", (2) Assets > Refresh menu, (3) `AssetDatabase.Refresh()` from code — auto-refresh is still FOCUS-gated in Unity 6; directory monitoring does not add background import.
- [U6000.0–U6000.x | MED | https://unity.com/releases/editor/whats-new/6000.3.0f1] No 6000.x release note found announcing background/unfocused auto-refresh; searches across 6000.0–6000.4 notes surface only unrelated asset-pipeline fixes (symlinks, cache-server) — treat "Unity 6 refreshes in background" as false absent new evidence.
- [U2022.1–U6000.x | HIGH | https://docs.unity3d.com/2022.3/Documentation/ScriptReference/EditorApplication-focusChanged.html] `EditorApplication.focusChanged` — "Raises when the Editor gets or loses focus in the operating system. EditorApplication.isFocused toggles before this event raises"; page exists for 2022.3 and 6000.0.
- [U2021.3–U2021.3 | MED | https://docs.unity3d.com/2021.3/Documentation/ScriptReference/EditorApplication-focusChanged.html] That URL 404s for 2021.3 — `focusChanged` is not in the public 2021.3 API surface; don't rely on it pre-2022.
- [U2021.3–U6000.x | LOW | https://discussions.unity.com/t/editor-recompiling-script-changes-even-while-in-background-is-this-new/903346] Reports of "Unity compiles in background by itself" trace to Rider's "Automatically refresh assets in Unity" plugin RPC, not to a Unity behavior change (thread resolved: disabling Rider setting stopped it; no Unity staff reply).

## (d) CRITICAL — is scripted `AssetDatabase.Refresh()` gated by prefs/focus?

- [U2019.3–U6000.x | HIGH | https://docs.unity3d.com/6000.0/Documentation/Manual/AssetDatabaseRefreshing.html] NO. Manual `AssetDatabase.Refresh()` is listed as an independent trigger, separate from (and not conditioned on) the focus/Auto-Refresh clause — a scripted call refreshes an unfocused editor.
- [U2021.3–U6000.x | HIGH | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/AssetDatabase.DisallowAutoRefresh.html] Strongest official wording: "This method does not influence the behaviour of AssetDatabase.Refresh. The Asset Database always performs a refresh if AssetDatabase.Refresh is called, regardless of this method and its internal counter" — even the explicit auto-refresh suppression API cannot block a manual Refresh.
- [U2021.3–U6000.x | HIGH | https://www.jetbrains.com/help/rider/Refreshing_Unity_Assets.html] Proof-by-product: Rider triggers asset refresh on save through its Unity editor plugin while Unity sits in the background (only exception documented: "Rider does not refresh assets if Unity is in the play mode"); it works even when Unity's own Auto Refresh pref is off.
- [U2021.3–U6000.x | LOW | https://discussions.unity.com/t/editor-auto-refreshes-when-auto-refresh-is-disabled/938010] Community corroboration: with Auto Refresh=Disabled, Visual Studio's "Refresh Unity's AssetDatabase on save" and VFX Graph "Always Refresh" still refresh the editor — the pref gates only the editor's own focus-triggered refresh, nothing programmatic.
- [U2021.3–U6000.x | LOW | https://discussions.unity.com/t/turn-off-auto-refresh-unity-still-auto-refreshes/1649233] Same pattern reported on recent versions: disabling Auto Refresh does not stop refreshes initiated by external tooling.
- Caveat: the call must execute on the editor main loop (remote command → main-thread dispatch). The unfocused editor still ticks enough to service such calls — that is exactly the Rider/VSTU mechanism above; no doc found claiming Refresh is deferred until focus when invoked from code.

## (e) Workarounds to refresh an unfocused editor

- [U2021.3–U6000.x | HIGH | https://www.jetbrains.com/help/rider/Refreshing_Unity_Assets.html] Preferred, documented approach: don't fake focus — invoke refresh in-process (editor-plugin RPC calling `AssetDatabase.Refresh()`), as Rider does on save/test-run.
- [U2018.2–U2019.4 | LOW | https://github.com/baba-s/unity-compile-in-background] Historic community tool "unity-compile-in-background" existed solely because Unity waits for focus; archived 2020 ("no longer needed — VS 2019 supports background compilation" via its own refresh-on-save RPC) — evidence the focus-wait was the pain point and programmatic refresh the cure.
- [U2019.1–U6000.x | LOW | https://forum.unity.com/threads/editor-compile-in-background.627952/] Community consensus thread: "you have to focus the editor to kick off recompilation" with window-activation as the naive workaround; no reliability guarantees stated.
- [U2021.3–U6000.x | MED | https://issuetracker.unity3d.com/issues/unity-editor-gets-focused-on-mac-when-recompiling-scripts-finishes-after-switching-windows-with-mac-mission-control] macOS focus<->reload coupling is real and buggy in reverse too: "Reload Script Assemblies" completion steals focus back to the Editor on Mac (Mission Control case) — focus juggling around reload is fragile on macOS.
- NOT FOUND: any authoritative (docs/staff) source documenting `osascript 'tell app "Unity" to activate'` or Win32 `SetForegroundWindow` as a refresh workaround or its failure modes — it exists only as folk practice; for a canonical reference, recommend in-process `AssetDatabase.Refresh()` and treat OS-level activation as unsupported.

## Per-version summary table

| Version | Pref key (type) | UI | Focus behavior |
|---|---|---|---|
| 2021.3 LTS | `kAutoRefresh` (bool) ≤~.9f1 → `kAutoRefreshMode` (int 0/1/2) from ~.10f1; `DirectoryMonitoring` (bool, Win-only) | Preferences > Asset Pipeline (docs still show checkbox) | auto-refresh only on focus regain when pref enabled |
| 2022.3 LTS | `kAutoRefreshMode` (0=Disabled, 1=Enabled, 2=EnabledOutsidePlaymode); legacy `kAutoRefresh` read as fallback | Preferences > Asset Pipeline, 3-value dropdown | same focus gate; `focusChanged` public API available |
| 6000.0/6000.3 | `kAutoRefreshMode` (unchanged); `DirectoryMonitoring` still Win-only | Edit > Preferences (macOS: Unity > Settings) > Asset Pipeline > Auto Refresh: Disabled / Enabled / Enabled Outside Playmode | still focus-gated; no background auto-import found in any 6000.x note |

**Bottom line for the MCP use-case (macOS, unfocused editor):** auto-refresh will not fire without focus on any version 2021.3→6000.x, and Directory Monitoring can't help (Windows-only, detection-only) — but a scripted `AssetDatabase.Refresh()` delivered into the editor process is documented to always refresh, unconditionally on prefs, focus, and even `DisallowAutoRefresh`.
