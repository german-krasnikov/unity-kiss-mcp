# T7 — Batch-mode / headless / out-of-process compile alternatives

## Ranked options (best → worst) for compile-checking WITHOUT the live interactive Editor

**Option 1 — Second Unity on a COPY of the project (the only fully correct out-of-process check)**
- [U2021.3–U6000 | HIGH | https://docs.unity3d.com/2021.3/Documentation/Manual/EditorCommandLineArguments.html] Hard blocker for same folder: "You can't open a project in batch mode while the Editor has the same project open; only a single instance of Unity can run at a time."
- [U2021.3–U6000 | HIGH | https://docs.unity3d.com/Manual/EditorCommandLineArguments.html] Identical single-instance wording persists in current (Unity 6) docs — restriction unchanged 2021.3→6000.x.
- [U2017–U6000 | MED | https://discussions.unity.com/t/multiple-unity-instances-cannot-open-the-same-project/607546] Lock = `Temp/UnityLockfile`; exact message: "It looks like another Unity instance is running with this project open. Multiple Unity instances cannot open the same project. If you are sure that there is no running Unity instance with the project open, remove .../Temp/UnityLockfile and restart Unity."
- [U2017–U6000 | MED | https://discussions.unity.com/t/it-looks-like-another-unity-instance-is-running-with-this-project-open-multiple-unity-instances-cannot-open-the-same-project/248925] In batch mode this is a HARD FAIL ("Aborting batchmode due to fatal error"), not a wait/retry; the process exits immediately.
- [U2017–U6000 | MED | https://discussions.unity.com/t/multiple-unity-instances-cannot-open-the-same-project/607546] Unity staff note: zombie `UnityShaderCompiler`/`JobProcess` children can keep the lock alive after a crash, causing false "already open" detection.
- [U5–U6000 | LOW | https://www.codedojo.com/?p=2143] Proven workaround: copy the whole project to a temp dir (it gets its own Library/Temp, so its own lock); multiple Unity instances on separate copies run fine in parallel on one machine/license.
- [U5–U6000 | MED | https://support.unity.com/hc/en-us/articles/115003118426-Running-multiple-instances-of-Unity-referencing-the-same-project] Unity Support has a dedicated article for "running multiple instances referencing the same project" (separate project directories per instance); same-directory sharing is unsupported.
- Cost caveat: a fresh copy pays full Library import once; subsequent runs on the kept copy are incremental.

**Option 2 — Headless compile-only invocation (when no Editor holds the project / on the copy)**
- [U2021.3–U6000 | HIGH | https://docs.unity3d.com/Manual/CommandLineArguments.html] macOS launch: `/Applications/Unity/Hub/Editor/<version>/Unity.app/Contents/MacOS/Unity -projectPath <path> ...`.
- [U2021.3–U6000 | HIGH | https://docs.unity3d.com/Manual/EditorCommandLineArguments.html] Cheapest recipe flags: `-batchmode -nographics -quit -logFile - -projectPath <p> -executeMethod <Static.NoOp>`; `-logFile -` streams the Editor log to stdout; `-nographics` skips GPU init (and disables logs unless `-logFile` given).
- [U2021.3–U6000 | HIGH | https://docs.unity3d.com/Manual/EditorCommandLineArguments.html] Exit contract: "When an exception occurs… or other operations fail, Unity immediately exits with return code 1"; `-executeMethod` errors should `throw` or call `EditorApplication.Exit(nonzero)`.
- [U2019–U6000 | MED | https://game.ci/docs/troubleshooting/common-issues/] Compile failure path: log shows "Scripts have compiler errors" (Argv.cpp) and the run aborts with exit code 1 — GameCI's quick-ref classifies "Scripts have compiler errors (exit code 1)" as a COMPILE failure.
- [U2019–U6000 | MED | https://gitlab.com/game-ci/unity3d-gitlab-ci-example/-/issues/116] With `-executeMethod`, compiler errors prevent the method from running → "Aborting batchmode due to failure" → nonzero exit; a no-op static method is the de-facto compile gate.
- [U2021.3–U6000 | HIGH | https://docs.unity3d.com/2021.3/Documentation/Manual/EditorCommandLineArguments.html] Pass `-accept-apiupdate`: "The APIUpdater doesn't run if you omit this… in batch mode. This might lead to compiler errors" that the interactive Editor wouldn't show.
- [U2019.4–U2022.1 | MED | https://issuetracker.unity3d.com/issues/unity-terminates-with-error-code-0-when-an-exception-occurs-while-importing-a-package-in-bach-mode] Don't trust exit code alone: known bug class where Unity exits 0 despite failure (package-import exception in batch mode; repro 2019.4–2022.1). Parse the log for "Scripts have compiler errors" too.
- [U2018.3 | LOW | https://discussions.unity.com/t/unity-2018-3-0b1-returns-exit-code-0-even-though-project-is-not-compiling/715610] Historical precedent of exit-0-despite-compile-errors with `-runEditorTests` — reinforces log-parsing as the source of truth.
- [U2022.3–U6000 | MED | https://docs.unity3d.com/Manual/ImporterConsistency.html] `-consistencyCheck` does NOT help script compile: it re-checks asset importer results against the cache and logs importer inconsistencies at startup — assets, not C#.
- [U2021.3–U6000 | HIGH | https://docs.unity3d.com/Packages/com.unity.test-framework@1.4/manual/reference-command-line.html] `-runTests -testPlatform EditMode` also forces compilation, but exit codes are explicitly undocumented: "There is currently no common definition for exit codes reported by individual Unity components under test."
- NOT FOUND: any Unity-6-specific flag for "compile scripts only, then exit" (no `-buildScripts`/compile-only argument exists in the 6000.x CLI reference).

**Option 3 — External Roslyn/csc/MSBuild against generated csproj or `Library/ScriptAssemblies` (fast, approximate)**
- [U2021.3–U6000 | MED | https://discussions.unity.com/t/expose-sln-and-csproj-generation/892301] Unity staff (HaraldNielsen): "Csproj's today are only for the IDE experience" — they are not what Unity actually compiles with; treat external builds as approximations.
- [U2021.3–U6000 | MED | https://discussions.unity.com/t/expose-sln-and-csproj-generation/892301] Staff blocker: "Unity needs to run at least once before to populate the Library folder… anything that depends on packages will fail to compile" — external compile requires a previously-imported project.
- [U2021.3–U6000 | LOW | https://discussions.unity.com/t/expose-sln-and-csproj-generation/892301] Working community pattern: launch Unity once to sync the .sln (`-executeMethod` solution-sync), quit, then run MSBuild-based analysis (used for SonarQube) — valid as a syntax/type check.
- [U2017–U6000 | MED | https://github.com/microsoft/MSBuildForUnity] MSBuildForUnity generates a VS solution "for building the Unity project into DLLs outside of Unity" per platform and InEditor/Player variants — the most structured external-compile route (Microsoft, low maintenance activity).
- CANNOT catch / must replicate:
- [U2021.3–U6000 | HIGH | https://docs.unity3d.com/2021.3/Documentation/Manual/ScriptCompilationAssemblyDefinitionFiles.html] Per-asmdef define scoping ("Symbols defined in the Assembly Definition are only in scope for the scripts in the assembly"), Define Constraints, Version Defines, platform include/exclude — raw csc must reproduce each assembly's exact define set or results diverge.
- [U2021.3–U6000 | HIGH | https://docs.unity3d.com/2021.3/Documentation/Manual/roslyn-analyzers.html] Source generators are "an additional step in your script compilation process" gated by the case-sensitive `RoslynAnalyzer` label and asmdef-folder scoping; csc without the same generator refs gives false missing-symbol errors (or misses generator failures).
- [U2020–U6000 | LOW | https://gio.blue/blog/28feb24-unity-ilpp] IL post-processing (`ILPostProcessor`) is an undocumented Unity compilation hook; external csc never runs it, so ILPP-injected code/failures (Netcode codegen etc.) are invisible.
- [U2021.3–U6000 | MED | https://github.com/needle-mirror/com.unity.burst/blob/master/Unity.Burst.CodeGen/BurstILPostProcessor.cs] Burst hooks compilation via its own `BurstILPostProcessor`; Burst compile errors are produced by the Burst compiler, not csc — undetectable externally.
- [U2021–U6000 | MED | https://discussions.unity.com/t/failed-to-copy-library-bee-artifacts-to-library-scriptassemblies-with-cp/928768] `Library/ScriptAssemblies` is the copy target of the last successful Bee compile — referencing it checks against last-good state, which goes stale the moment in-editor code changes.

**Option 4 — Poking `Library/Bee` directly: NOT viable as an interface**
- [U2021–U6000 | MED | https://blog.unity.com/engine-platform/accelerating-player-builds-with-incremental-build-pipeline] Architecture: a .NET "build program" (one exists specifically for script compilation) generates a build graph into a `.dag.json`; `bee_backend` reads it, diffs changed inputs, and spawns the Roslyn compile processes.
- [U2021–U6000 | MED | https://github.com/Unity-Technologies/UnityCsReference/blob/master/Editor/Mono/Scripting/ScriptCompilation/EditorCompilation.cs] Editor compilation drives Bee internally (TundraBuildState.state, bee_backend.info bookkeeping) — entry points are Editor-internal, not CLI-exposed.
- [U2021–U6000 | LOW | https://git.cardiff.ac.uk/c21013377/unity-pong/-/tree/master/Library/Bee/artifacts/1900b0aE.dag] `Library/Bee/artifacts/<hash>.dag/` contains per-assembly outputs and response files — parsable for forensics (rsp = exact csc args), but layout is unversioned/undocumented.
- [U2021–U6000 | LOW | https://forum.unity.com/threads/persistent-error-internal-build-system-error-backend-exited-with-code-2.1368849/] Unity itself labels backend failures "Internal build system error" — Bee/bee_backend is internal; no supported standalone invocation.
- NOT FOUND: any documented/supported way to invoke ScriptCompilationBuildProgram or bee_backend standalone for a compile check.

**Option 5 — CI services (what they actually do)**
- [U2019–U6000 | MED | https://game.ci/docs/github/builder/] game-ci has NO compile-only mode: unity-builder runs a full `BuildPipeline` build (custom `-executeMethod` allowed); compile errors fail the job via the batch-mode exit-1 path.
- [U2019–U6000 | MED | https://game.ci/docs/github/test-runner/] Cheapest game-ci compile gate in practice: unity-test-runner with EditMode — the test run forces a full script compile before any test executes.
- [U2019–U6000 | MED | https://game.ci/docs/troubleshooting/common-issues/] game-ci compile-failure triage: clear `Library/ScriptAssemblies`, check for unhydrated git-LFS pointer files and accidentally-ignored sources.
- [U2021.3–U6000 | MED | https://support.unity.com/hc/en-us/articles/46719618344852-Why-is-my-Unity-Build-Automation-build-failing] Unity Build Automation runs full builds in batch mode on clean virtual builders; "Player Export Failed" with script compilation errors is its most common failure — diagnosis is log-based, no pre-build compile-only stage.
- NOT FOUND: a dedicated "validate compilation only" job type in Unity Build Automation or game-ci documentation.

## Bottom line for the parent agent
- Same project folder + live Editor → second Unity process is a guaranteed hard fail (UnityLockfile), batch mode aborts with exit 1; never a wait. Only escape: separate copy (or rsync/worktree) with its own Library.
- Canonical cheap check on a free project copy: `Unity -batchmode -nographics -accept-apiupdate -quit -logFile - -projectPath <copy> -executeMethod CI.NoOp` → exit 0 = compiles; grep log for "Scripts have compiler errors" as belt-and-suspenders (known exit-0 bugs).
- External csc/MSBuild = fast pre-filter only: misses source generators, ILPP/Burst, per-asmdef defines, package resolution; Unity staff state csproj is IDE-only.
- `Library/Bee` is parsable (rsp/dag.json) but internal and unsupported; do not build tooling on it.
