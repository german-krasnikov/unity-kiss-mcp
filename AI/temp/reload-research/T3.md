# T3 — Domain Reload Mechanics & Editor State Lifecycle

## (a) What a domain reload IS + full phase timeline

- [U2021.3–U6000.x | HIGH | https://docs.unity3d.com/6000.0/Documentation/Manual/domain-reloading.html] A domain is "an isolated section of memory dedicated to a particular application, which contains the compiled types ... grouped into assemblies" plus "the data that represents the current application state"; a domain reload tears down and recreates this.
- [U6000.5–U6000.x | HIGH | https://docs.unity3d.com/6000.5/Documentation/Manual/programming-code-lifecycle.html] Canonical 5-phase code-reload lifecycle: 1) recompile script assemblies → 2) serialize managed objects → 3) unload/destroy existing assemblies → 4) load new assemblies → 5) deserialize managed objects and initialize.
- [U6000.5–U6000.x | HIGH | https://docs.unity3d.com/6000.5/Documentation/Manual/programming-code-lifecycle.html] Unity 6.5+ adds symmetric lifecycle attributes — `[OnCodeDeinitializing]` (during serialize), `[OnCodeUnloading]` (before unload), `[OnCodeLoaded]` (after load), `[OnCodeInitializing]` (after deserialize) — documented as the modern equivalents of `beforeAssemblyReload`/`[InitializeOnLoad]`/`[InitializeOnLoadMethod]`.
- [U2021.3–U6000.x | HIGH | https://docs.unity3d.com/2022.3/Documentation/Manual/ConfigurableEnterPlayModeDetails.html] Docs-listed in-reload order — before unload: `AssemblyReloadEvents.beforeAssemblyReload` → `OnDisable()` on all ScriptableObjects/MonoBehaviours → `OnBeforeSerialize()` → Mono domain unload (`AppDomain.DomainUnload`); after load: `OnAfterDeserialize()` → `OnValidate()` → `[ExecuteInEditMode]` lifecycle → `[InitializeOnLoad]` + `[InitializeOnLoadMethod]` → `AssemblyReloadEvents.afterAssemblyReload`.
- [U2021.3–U6000.x | HIGH | https://docs.unity3d.com/ScriptReference/InitializeOnLoadAttribute.html] `[InitializeOnLoad]` static ctors run on every recompile ("also known as a Domain Reload"), on editor launch, and on play-enter only if domain reload is enabled; they run BEFORE asset import completes, so asset loads there can return null — docs say use `AssetPostprocessor.OnPostprocessAllAssets` for asset work.
- [U2021.3–U6000.x | HIGH | https://docs.unity3d.com/ScriptReference/AssemblyReloadEvents-afterAssemblyReload.html] `afterAssemblyReload` "is dispatched just after Unity have reloaded all assemblies", on script changes and on entering play mode.
- [U2021.3–U6000.x | HIGH | https://docs.unity3d.com/ScriptReference/Callbacks.DidReloadScripts.html] `[DidReloadScripts]` notifies after script reload and is the only one with an explicit order index ("Builtin callbacks always have a value of 0").
- [U2021.3–U6000.x | LOW | https://discussions.unity.com/t/initializeonload-vs-didreloadscripts/572660] Community-tested: `InitializeOnLoad`/`InitializeOnLoadMethod` always run before `[DidReloadScripts]`; no Unity-staff confirmation in thread.
- NOT FOUND: official docs-level statement of `[DidReloadScripts]` position relative to `afterAssemblyReload` (only the community ordering above; the 6000.6 execution-order diagram at https://docs.unity3d.com/6000.6/Documentation/Manual/configurable-enter-play-mode-details.html is an image, list not in page text).
- [U2019.3–U6000.x | LOW | https://johnaustin.io/articles/2020/domain-reloads-in-unity] Community phase-cost breakdown: import changes → compile → "Swap the AppDomain" (dominated by serializing all managed objects, linear in object count) → `InitializeOnLoad` callbacks → misc (TypeCache regen ~300ms, UI repaint ~200ms).

## (b) isCompiling vs isUpdating — the "false gap" verdict

- [U2021.3–U6000.x | HIGH | https://docs.unity3d.com/ScriptReference/EditorApplication-isCompiling.html] `isCompiling` = "editor currently compiling scripts" (read-only), nothing more.
- [U2021.3–U6000.x | HIGH | https://docs.unity3d.com/ScriptReference/EditorApplication-isUpdating.html] `isUpdating` = "True if the Editor is currently refreshing the AssetDatabase" (file-change check + reimports), not compilation.
- VERDICT: the false gap is REAL, with three sourced mechanisms:
- [U2021.3–U6000.x | HIGH | https://docs.unity3d.com/Manual/AssetDatabaseRefreshing.html] Refresh phase order is: update DB → import/compile code files → "Reload the scripting domain, if Refresh was not invoked from a script" — i.e. when Refresh is called FROM script the domain reload is deferred past the refresh, so flags can read false while a reload is still pending.
- [U2019.3–U6000.x | HIGH | https://issuetracker.unity3d.com/issues/editorapplication-dot-iscompiling-is-not-called-when-the-scripts-are-recompiled-upon-refocusing-the-editor] "By Design": with ADBv2, refocus-triggered compilation runs synchronously and `isCompiling` never reflects it; Unity's official advice is to use compilation-started/finished events instead of polling `isCompiling`.
- [U2021.3–U6000.x | LOW | https://discussions.unity.com/t/how-to-wait-for-unity-to-compile-generated-script-while-running-editor-script/231945] After generating a script + Refresh, compilation/reload happens on a later editor tick and kills coroutines/continuations mid-wait; the working pattern is not waiting at all but resuming via `[DidReloadScripts]` with state parked in EditorPrefs.
- [U2021.3–U6000.x | MED | https://discussions.unity.com/t/how-to-wait-for-domain-reload-in-code/929827] Reliable "fully idle" practice per community thread: don't poll flags — subscribe `AssemblyReloadEvents.afterAssemblyReload` / use `InitializeOnLoad(Method)` and treat reload-completion as an event, re-subscribing each domain (OnEnable/OnDisable pairing).

## (c) Threads, TCP sockets, SynchronizationContext (plugin-critical)

- [U2021.3–U6000.x | HIGH | https://learn.microsoft.com/en-us/dotnet/api/system.appdomain.unload?view=net-5.0] AppDomain unload terminates the domain's threads via `Thread.Abort` → `ThreadAbortException` raised in each; finally blocks still run and can delay unload unpredictably (Mono/.NET Framework semantics, which Unity's editor Mono follows).
- [U2021.3–U6000.2 | HIGH | https://issuetracker.unity3d.com/issues/editor-is-frozen-on-reloading-domain-when-entering-play-mode-for-the-second-time-using-socket-dot-poll-1-dot-dot-dot] WON'T FIX (repro'd 2021.3.51f1, 2022.3.61f1, 6000.0.48f1, 6000.1.0f1, 6000.2.0a10): a thread blocked in `Socket.Poll(-1)` sits in the OS-native `select` "with no way to exit the wait" → editor freezes on "Reloading Domain"; Unity's position: threads must COOPERATIVELY end on domain reload, forceful termination is unsafe.
- [U2021.3–U6000.x | HIGH | https://issuetracker.unity3d.com/issues/editor-gets-stuck-on-reloading-domain-when-re-entering-play-mode-after-starting-a-named-pipe-without-closing-it] Same failure class for an unclosed named pipe: editor stuck on "Reloading Domain" — any blocking IPC handle left open across reload can hang the unload.
- [U2021.3–U6000.x | HIGH | https://docs.unity3d.com/ScriptReference/AssemblyReloadEvents-beforeAssemblyReload.html] `beforeAssemblyReload` ("raised before the C# domain is reloaded") is the designated hook to close listeners/sockets and join threads before unload.
- [U2021.3–U6000.x | LOW | https://github.com/CoplayDev/unity-mcp/issues/1173] Real-world MCP precedent: TcpListener leaked across domain reload (port not released in time on Windows; ~500ms wait too short; silent port-fallback masked it) — OS socket handles outlive the managed domain if not explicitly closed.
- [U6000.0–U6000.x | HIGH | https://docs.unity3d.com/6000.3/Documentation/Manual/async-awaitable-continuations.html] "Unity overwrites the default `SynchronizationContext` with a custom `UnitySynchronizationContext`"; main-thread continuations run "on the next frame Update tick on the main thread"; and "Unity doesn't automatically stop code running in the background when you exit Play mode" (use `Application.exitCancellationToken`).
- [U2021.3–U6000.x | MED | https://github.com/Unity-Technologies/UnityCsReference/blob/master/Runtime/Export/Scripting/UnitySynchronizationContext.cs] The sync context is (re)initialized on initial domain load and on domain reload (MonoManager ReloadAssembly); its work queue is managed state — posts queued in the old domain are discarded with it, never executed in the new domain (speculative on the "never executed" wording, but queue is per-domain managed memory).
- Practical consequence for our TCP listener: close socket + signal/join the accept thread in `beforeAssemblyReload` (cooperative exit; never rely on ThreadAbort reaching a thread parked in native `select`/`Accept`), then rebind in `[InitializeOnLoad]`/`afterAssemblyReload`.

## (d) Cross-reload state: SessionState vs EditorPrefs vs disk

- [U2021.3–U6000.x | HIGH | https://docs.unity3d.com/ScriptReference/SessionState.html] `SessionState` = "Key-Value Store intended for storing and retrieving Editor session state that should survive assembly reloading"; "cleared when Unity exits"; docs explicitly point to EditorPrefs for cross-session persistence.
- [U2021.3–U6000.x | HIGH | https://docs.unity3d.com/ScriptReference/EditorPrefs.html] `EditorPrefs` persists per-machine across editor sessions AND across projects (macOS `~/Library/Preferences/com.unity3d.UnityEditor5.x.plist`, Windows `HKCU\Software\Unity Technologies\Unity Editor 5.x`, Linux `~/.local/share/unity3d/prefs`); null strings stored as empty string.
- [U2021.3–U6000.x | MED | https://docs.unity3d.com/Manual/AssetDatabaseRefreshing.html] Files on disk obviously survive everything, but writing assets mid-handshake re-triggers refresh/compile phases — write outside `Assets/` (e.g. `Library/`, `Temp/`) for reload handshakes to avoid recursive refresh.
- Handshake guidance: SessionState = per-editor-run reload survival (right tool for "was my listener running before reload"); EditorPrefs = machine-global durable config (wrong for per-project per-run flags — leaks across projects); disk = cross-process/cross-session.

## (e) Enter Play Mode Options vs edit-mode reloads

- [U2021.3–U6000.x | HIGH | https://docs.unity3d.com/6000.0/Documentation/Manual/domain-reloading.html] Verdict: EPMO "Reload Domain off" affects ONLY play-mode-enter reloads — "Unity also performs domain reload as part of an asset database refresh when it detects changes to scripts. This still happens even when domain reload on entering Play mode is disabled." Script-change edit-mode reloads are untouched.
- [U2021.3–U6000.x | HIGH | https://docs.unity3d.com/6000.0/Documentation/Manual/domain-reloading.html] With domain reload disabled, statics and static event subscribers persist across play sessions (must be reset manually); with it enabled they reset to defaults each reload.
- [U2021.3–U6000.x | HIGH | https://docs.unity3d.com/ScriptReference/InitializeOnLoadAttribute.html] Corroboration: `[InitializeOnLoad]` fires on play-enter only "if domain reload is enabled", but unconditionally on script recompile.

## Key takeaways for our TCP plugin

1. Reload survival sequence to rely on (docs-backed): `beforeAssemblyReload` → OnDisable → OnBeforeSerialize → domain swap → OnAfterDeserialize/OnValidate → InitializeOnLoad(Method) → `afterAssemblyReload`; `[DidReloadScripts]` after InitializeOnLoad (community-only ordering).
2. The isCompiling/isUpdating both-false gap is confirmed by three independent sourced mechanisms (deferred reload on scripted Refresh, By-Design sync compile on refocus, next-tick compile start) — polling alone can never prove "editor idle"; event-driven handshake via SessionState + afterAssemblyReload is the only docs-aligned pattern.
3. Thread/socket fate: managed threads get ThreadAbortException at unload, but threads blocked in native socket waits are unkillable and hang the editor (Unity Won't Fix, repro through 6000.2) — cooperative shutdown in `beforeAssemblyReload` is mandatory, and queued SynchronizationContext posts die with the old domain.
