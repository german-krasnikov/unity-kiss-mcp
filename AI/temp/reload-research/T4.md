# T4 — UPM local `file:` package update mechanics (sourced verdict)

## (a) Is a `file:` package file-watched? What triggers re-import of an edited .cs?

- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.3/Documentation/Manual/upm-concepts.html] Local-folder (`file:`) packages are **mutable and used in place** — they are NOT copied to the global cache (unlike registry/git/tarball packages); your edits modify the very files Unity loads.
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.3/Documentation/Manual/AssetDatabaseRefreshing.html] An Asset Database refresh checks "if any files in the Assets **and Packages** folders have been added, modified, or deleted" — local `file:` packages are mounted under the virtual `Packages/` folder, so .cs edits in them are picked up by the **normal refresh path**, not by package resolution.
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.3/Documentation/Manual/AssetDatabaseRefreshing.html] Refresh triggers are exactly: (1) Editor regains focus **only if Auto-Refresh preference is enabled**, (2) menu Assets > Refresh, (3) `AssetDatabase.Refresh()` from code. Nothing else; there is no independent package file-watcher.
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.3/Documentation/Manual/preferences-asset-pipeline.html] "Directory Monitoring" (watch directories instead of scanning all files) is **Windows only** — on macOS there is no OS-level file watching; change detection happens only at refresh time via scan. This is the root of the "edits not picked up until focus/restart" pain on Mac.
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.3/Documentation/Manual/preferences-asset-pipeline.html] Auto Refresh has three states (Disabled / Enabled / Enabled Outside Playmode) — if Disabled or you're in Play mode with the third option, focus does nothing; only manual Refresh works.

## (b) `Client.Resolve()`, `Client.Add()`, `Events.registeredPackages`

- [U2021.3–U6000.2 | HIGH | https://docs.unity3d.com/2021.3/Documentation/ScriptReference/PackageManager.Client.Resolve.html] `public static void Resolve()` exists and is identically documented across 2021.3 → 6000.2: "Forces the Package Manager to resolve the project's packages, reinstalling any altered or missing package and removing extraneous packages."
- [U2021.3–U6000.2 | HIGH | https://docs.unity3d.com/6000.2/Documentation/ScriptReference/PackageManager.Client.Resolve.html] Resolve is fire-and-forget void (results arrive via `Events.registeringPackages`/`registeredPackages`); "If the packages are already up-to-date, **no event is raised**" — i.e. for an in-place local package whose version did NOT change, Resolve is documented as a no-op for content; docs say nothing about it reimporting unchanged local files. Also: "Make sure any other Client operations complete before calling this method."
- [U2020.2–U6000.3 | HIGH | https://issuetracker.unity3d.com/issues/packman-isnt-refreshed-when-calling-assetdatabase-dot-refresh-after-making-changes-to-a-pacakge] Unity's official "By Design" resolution (issue 1248326, regressed in 2020.2): `AssetDatabase.Refresh()` does not update package *registration/metadata*; "if only package metadata changed, replace AssetDatabase.Refresh() with Client.Resolve(); if the scope of changes is unknown, call **both**." This is Unity's canonical division of labor: Refresh = file content, Resolve = manifest/metadata.
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/ScriptReference/PackageManager.Client.Add.html] `Client.Add(string)` is async (returns `AddRequest`, poll `Status` in update loop), accepts `file:/path/to/folder` identifiers; docs do not define behavior when the same package is already installed.
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.3/Documentation/ScriptReference/PackageManager.Events-registeredPackages.html] `Events.registeredPackages` fires "after applying changes to the registered packages list… after the asset database refresh completes, which occurs after scripts compile and the domain reloads"; handlers are wiped by the domain reload, so register via `[InitializeOnLoadMethod]`.
- NOT FOUND: official statement of the exact Unity version where `Client.Resolve` was introduced (2021.3 docs confirm presence throughout our target range; the OpenUPM 2020.1 round-up does not mention it).

## (c) `packages-lock.json` and local `file:` deps

- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.3/Documentation/Manual/upm-conflicts-auto.html] The lock file stores dependency-resolution results (version, depth, source, dependencies, url); "To force a refresh of indirect dependency versions, **delete the lock file**"; `enableLockFile: false` in manifest disables it (at the cost of re-cloning git packages and non-determinism); never hand-edit it — UPM overwrites changes.
- [U2020.1–U6000.3 | MED | https://medium.com/openupm/unity-package-manager-2020-1-round-up-54a1d32c60fc] `packages-lock.json` was introduced in Unity 2020.1 (npm/yarn-style lock).
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.3/Documentation/Manual/upm-concepts.html] The global UPM cache does NOT apply to local-folder packages (used in place); it DOES apply to local **tarballs**, which are extracted into the cache and immutable — so `file:*.tgz` deps go stale on rebuild, `file:folder` deps cannot.
- [U2019.4–U6000.3 | MED | https://discussions.unity.com/threads/packages-lock-json.912602/] Unity staff (Adrian): lock exists for reproducibility ("same commit is used for git packages"); deleting it is safe — "delete the lock and Unity will re-download everything and re-generate the lock."
- NOT FOUND: any doc that the lock pins a content hash for local `file:` folder packages (docs describe hash/lock semantics only for git; local folder content is never hashed/pinned per available docs — deleting the lock is therefore irrelevant to stale .cs content in a `file:` folder dep).

## (d) Embedded vs `file:` reference

- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.3/Documentation/Manual/upm-embed.html] Embedded package = "a mutable package that you store under the `Packages` directory at the root of a Unity project"; "the package on disk takes priority over the version of the package listed as a dependency"; embedding methods: copy folder into `Packages/`, Package Manager **Customize**, or `Client.Embed` via script.
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.3/Documentation/Manual/cus-location.html] Official comparison: embedded = implicit (no manifest entry, "Unity automatically detects packages in the Packages subfolder"), label "Custom", same repo as project, highly portable; external `file:` = explicit manifest reference, label "Local", limited portability. Unity does NOT declare either one superior for active development — choice is contextual.
- [U2021.3–U6000.3 | MED | https://docs.unity3d.com/6000.3/Documentation/Manual/upm-concepts.html] Both embedded and local-folder are the only two mutable sources; refresh/import mechanics are the same scan of the `Packages` mount — the docs document NO refresh-behavior difference between them. Embedded's practical edge is only that files physically live inside the project root (one watched/scanned tree, no cross-root path).
- NOT FOUND: any official statement that embedding fixes "edits not picked up" vs `file:` reference (no docs/issuetracker source claims a detection difference).

## (e) "Bump package.json version" trick — documented or folklore?

- NOT FOUND: any docs.unity3d.com / issuetracker statement that bumping the local package's `version` field forces re-import or is required for content pickup. **Verdict: folklore.** Mechanically it works only because changing `package.json` alters package metadata → next resolve sees an "altered" package → registration change → `registeredPackages` → full refresh+domain reload, per the Resolve docs above.
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.3/Documentation/Manual/upm-concepts.html] What actually "invalidates" a local folder package: nothing needs invalidating — content is loaded in place; only metadata (`package.json`) changes go through the resolver. (mtime/manifest-hash invalidation for local folders: NOT FOUND in any official source.)
- [U2019.1–U6000.3 | LOW | https://discussions.unity.com/threads/solved-force-reload-package.629140/] Unity staff (maximeb_unity) note the catch-22: "all compilation errors must be resolved before it reloads the active assemblies" — if the consuming project has compile errors, NO mechanism (Refresh/Resolve/bump) will load your fixed local-package code until errors clear; the editor stays on stale assemblies.

## VERDICT — minimal reliable action for an edited .cs in a `file:` package

1. **Content-only edit (.cs body): `AssetDatabase.Refresh()`** (or editor focus with Auto Refresh=Enabled) is the documented and sufficient trigger — the file is in place, the Packages mount is scanned. Resolve/lock-delete/version-bump add nothing for this case. [HIGH — AssetDatabaseRefreshing.html + upm-concepts.html]
2. **Metadata edit (package.json, .asmdef add/remove, deps): `Client.Resolve()` + `AssetDatabase.Refresh()`** — Unity's official "unknown scope → call both" guidance. [HIGH — issuetracker 1248326]
3. **On macOS specifically**, never expect background pickup: no Directory Monitoring (Windows-only), so between focus events the editor is blind. Programmatic `AssetDatabase.Refresh()` over your TCP bridge is strictly more reliable than `osascript activate` (focus only works when Auto Refresh enabled and not gated by Play mode). [HIGH — preferences-asset-pipeline.html]
4. Version bump is a heavyweight folklore substitute for `Client.Resolve()`; lock-file deletion is for indirect/git version drift, irrelevant to local folder content.
