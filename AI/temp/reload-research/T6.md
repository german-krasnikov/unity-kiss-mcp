# T6 — LockReloadAssemblies / DisallowAutoRefresh: pairing rules & failure modes

## (a) What each blocks; pairing guidance

- [U2021–U6000 | HIGH | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/EditorApplication.LockReloadAssemblies.html] `LockReloadAssemblies` "prevents loading of assemblies when it is inconvenient" — it blocks **assembly (domain) reload only**, e.g. to preserve state during drag operations; Unity itself auto-locks during mouse-down.
- [U2021–U6000 | HIGH | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/AssetDatabase.DisallowAutoRefresh.html] `DisallowAutoRefresh` "increments an internal counter which Unity uses to determine whether to allow automatic AssetDatabase refreshing behavior" — it blocks **automatic asset-database refresh** (file scan + import), not assembly reload per se.
- [U2021–U6000 | HIGH | https://docs.unity3d.com/2021.3/Documentation/ScriptReference/AssetDatabase.DisallowAutoRefresh.html] Same text verified on the 2021.3 page, incl.: "This method does not influence the behaviour of AssetDatabase.Refresh. The Asset Database always performs a refresh if AssetDatabase.Refresh is called, regardless of this method and its internal counter."
- [U2021–U6000 | HIGH | https://docs.unity3d.com/6000.0/Documentation/Manual/AssetDatabaseRefreshing.html] Refresh pipeline order: scan changed files → "Import and compile code-related files such as .dll, .asmdef, .asmref, .rsp, and .cs files" → "Reload the scripting domain, if Refresh was not invoked from a script" → post-process → hot reload. So refresh = import+compile; domain reload is a separate, conditional final step.
- NOT FOUND: any official docs.unity3d.com guidance explicitly telling you to *pair* LockReloadAssemblies with DisallowAutoRefresh. Pairing is a community pattern only.
- [U2021–U6000 | LOW | https://discussions.unity.com/t/can-i-stop-auto-compile-after-edit-create-or-remove-a-script/878449] Community pattern: call `LockReloadAssemblies()` to block recompil-reload and `DisallowAutoRefresh()` to block auto-refresh — used together to fully freeze the compile/reload cycle.

## (b) Ref-counting and unbalanced calls

- [U2021–U6000 | HIGH | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/AssetDatabase.AllowAutoRefresh.html] Disallow/Allow are explicitly ref-counted: "Each pair increments and decrements the counter by one, and if your code is correctly nested, the final outer call to AllowAutoRefresh sets the counter to zero" — nesting is safe by design.
- [U2021–U6000 | HIGH | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/AssetDatabase.AllowAutoRefresh.html] Over-release is documented as broken: "Your code should never cause the counter to go below zero. This method asserts when the internal counter goes below zero but still continues to decrement it" — i.e. extra AllowAutoRefresh calls permanently wedge auto-refresh until counter is re-balanced.
- [U2021–U6000 | HIGH | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/EditorApplication.LockReloadAssemblies.html] Lock side is implicitly counted via pairing rule: "Each LockReloadAssemblies must be matched by UnlockReloadAssemblies, otherwise scripts will never unload." No documented assert for over-unlock on this API.
- [U2020–U2020 | MED | https://github.com/inkle/ink-unity-integration/issues/137] Real-world unbalance symptom: `Assertion failed on expression: 'm_DisallowAutoRefresh >= 0'` thrown from `AssetDatabase:AllowAutoRefresh()` (ink plugin, Unity 2020.1.12) — the assert text confirms the counter lives natively as `m_DisallowAutoRefresh`.
- [U2021–U6000 | MED | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/EditorApplication.LockReloadAssemblies.html] Exception between lock and unlock = unlock never runs = "scripts will never unload" for the rest of the session; docs offer no auto-expiry or timeout. try/finally is mandatory (inference from the pairing rule; no official statement on exceptions).

## (c) Survival across domain reload / crash; detection & recovery

- [U2021–U6000 | MED | https://github.com/Unity-Technologies/UnityCsReference/blob/master/Editor/Mono/EditorApplication.bindings.cs/] Both calls are `public static extern` bound to native `GetApplication()` (`[StaticAccessor("GetApplication()", StaticAccessorType.Dot)]`) — counters live in **native editor state**, not managed statics, so a domain reload does not reset them (inference from binding location; flagged).
- [U2021–U2022 | HIGH | https://issuetracker.unity3d.com/issues/domain-reload-missing-when-entering-play-mode] Corroboration that the lock persists into reload decision points: in 2021.2.0b12/2022.1.0a7-a8 the engine "checked LockReloadAssemblies when doing domain reload before play mode and would cause domain reload to be skipped if set" (fixed in 2021.2.X/2022.1.X) — i.e. a held lock can silently swallow even play-mode domain reloads in affected versions.
- NOT FOUND: official statement on whether the counters survive an editor crash/restart. (Native in-memory counters logically reset with the process — flagged speculation, no URL claims process persistence.)
- [U2021–U6000 | MED | https://github.com/Unity-Technologies/UnityCsReference/blob/master/Editor/Mono/EditorApplication.bindings.cs/] No public query API for lock depth. The only probe is `internal static extern bool CanReloadAssemblies()` on `EditorApplication` — reflection-only, boolean, no depth.
- NOT FOUND: any public API to read the DisallowAutoRefresh counter value (nothing in docs; no `IsAutoRefreshDisabled` in the Scripting API index).
- [U2021–U6000 | MED | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/EditorUtility.RequestScriptReload.html] Recovery lever once balanced: `EditorUtility.RequestScriptReload()` — "The Unity Editor reloads script assemblies asynchronously on the next frame" (no recompilation), usable to force the missed reload after a stuck lock is released.
- [U2020–U2021 | MED | https://issuetracker.unity3d.com/issues/code-coverage-windows-adding-new-included-slash-excluded-paths-row-locks-assembly-reload-in-the-editor] Stuck-lock signature & manual recovery: lock icon at editor bottom-right, "Scripts do not recompile"; workaround "Perform any operation that unlocks assembly reload (for example, drag a window)" — confirms Unity's own UI flows can leave the same native lock held (dup of issue #1367358, OpenFolderPanel on Windows, 2020.3.0f/2021.1.0b10).

## (d) AssetDatabase.Refresh while LockReloadAssemblies is held

- [U2021–U6000 | HIGH | https://docs.unity3d.com/2021.3/Documentation/ScriptReference/AssetDatabase.DisallowAutoRefresh.html] Explicit `Refresh()` always runs the refresh regardless of DisallowAutoRefresh ("The Asset Database always performs a refresh if AssetDatabase.Refresh is called") — so neither API stops an explicit Refresh's import work.
- [U2021–U6000 | HIGH | https://docs.unity3d.com/6000.0/Documentation/Manual/AssetDatabaseRefreshing.html] Within a refresh, domain reload is a separate conditional step ("Reload the scripting domain, if Refresh was not invoked from a script") — script-invoked Refresh imports and compiles but defers the reload anyway.
- [U2021–U6000 | HIGH | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/AssetDatabase.Refresh.html] Refresh "happens synchronously for asset imports, and asynchronously for script compilation" — imports complete under the lock; compilation is queued async; only the reload is what the lock can block.
- [U2021–U6000 | MED | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/EditorApplication.LockReloadAssemblies.html] Net behavior under lock: import done immediately, compile proceeds, **reload queued until UnlockReloadAssemblies** — direct consequence of "scripts will never unload" until unlock (composition of the three doc statements; no single official sentence states it).
- NOT FOUND: official doc on the exact moment the queued reload fires after Unlock (same call stack vs next editor tick). Community evidence says it is not always immediate:
- [U2021–U2023 | LOW | https://discussions.unity.com/t/assetdatabase-allowautorefresh-not-working/927783] After re-allowing, automatic behavior did not resume by itself; an explicit `AssetDatabase.Refresh()` right after `AllowAutoRefresh()` was needed to kick the pipeline (user-confirmed workaround, no staff reply).
- [U2021–U6000 | LOW | https://issuetracker.unity3d.com/issues/code-coverage-windows-adding-new-included-slash-excluded-paths-row-locks-assembly-reload-in-the-editor] Pending reload fires on the next editor interaction/tick once the lock is released (window drag sufficed) — consistent with "deferred until unlock + next pump", not synchronous inside Unlock.

## (e) Known bugs 2021–6000 (plus load-bearing older ones)

- [U2021–U2022 | HIGH | https://issuetracker.unity3d.com/issues/domain-reload-missing-when-entering-play-mode] Held LockReloadAssemblies skipped the play-mode domain reload entirely (empty-project repro; fixed 2021.2.X / 2022.1.X).
- [U2021–U6000 | HIGH | https://issuetracker.unity3d.com/issues/auto-refresh-is-still-active-when-its-set-to-to-disable-in-the-preferences] UUM-40547: Auto Refresh ran despite "Disable" preference (2021.3.27f1, 2022.3.3f1, 2023.x, 6000.0.0b11; fixed 2021.3.44f1 / 2022.3.X / 6000.0.X) — the preference, and by extension refresh-suppression assumptions, were unreliable across this whole range.
- [U2020–U2021 | MED | https://issuetracker.unity3d.com/issues/code-coverage-windows-adding-new-included-slash-excluded-paths-row-locks-assembly-reload-in-the-editor] Windows: OpenFolderPanel/Code-Coverage UI left assembly-reload lock held; compilation silently stopped until manual UI interaction.
- [U2020–U2020 | MED | https://github.com/inkle/ink-unity-integration/issues/137] `m_DisallowAutoRefresh >= 0` assertion from third-party unbalanced AllowAutoRefresh; fixed plugin-side in ink 0.9.7 (per https://github.com/inkle/ink-unity-integration/issues/145).
- [U5–U5 | MED | https://issuetracker.unity3d.com/issues/lockreloadassemblies-is-not-working-correctly] Historical: lock failed to prevent reload on focus switch (fixed 5.4) — precedent that the lock is not absolute in buggy versions.
- [U2021–U2021 | LOW | https://discussions.unity.com/t/editor-auto-refreshes-when-auto-refresh-is-disabled/938010] External triggers bypass refresh suppression: Visual Studio "Refresh Unity's AssetDatabase on save" and VFX-Graph "Always Refresh" force refreshes independent of Unity-side settings.

## Safe usage pattern (sourced synthesis)

- [U2021–U6000 | HIGH | https://docs.unity3d.com/6000.0/Documentation/ScriptReference/AssetDatabase.AllowAutoRefresh.html] Pattern: `DisallowAutoRefresh(); LockReloadAssemblies(); try { ... } finally { UnlockReloadAssemblies(); AllowAutoRefresh(); }` — strict 1:1 pairing per the documented counter contracts; never let Allow outnumber Disallow (documented assert + negative counter).
- [U2021–U2023 | LOW | https://discussions.unity.com/t/assetdatabase-allowautorefresh-not-working/927783] After releasing, call `AssetDatabase.Refresh()` once to flush anything that accumulated while suppressed.
- [U2021–U6000 | MED | https://github.com/Unity-Technologies/UnityCsReference/blob/master/Editor/Mono/EditorApplication.bindings.cs/] Reload-survival caveat: the native counters outlive your managed state. If your guard object is destroyed by a domain reload (e.g. play-mode enter, package import) without running `finally`, the lock stays held with no managed owner — keep an out-of-band marker (e.g. SessionState/EditorPrefs flag) and re-balance on `[InitializeOnLoad]`, probing `EditorApplication.CanReloadAssemblies()` via reflection.

## Verdict for our ReloadGuard (lock without DisallowAutoRefresh)

- Not fatal, but leaky: per (a)/(d), during a long AI turn auto-refresh still scans/imports assets and queues async compilation; only the reload is held. Unlock at turn-end then releases a reload that fires on the next pump — that part is fine for us.
- Real risks found: (1) unbalanced/lost unlock after a play-mode or externally-forced domain reload — native counter survives, managed guard doesn't ([U2021–U6000 | MED | https://github.com/Unity-Technologies/UnityCsReference/blob/master/Editor/Mono/EditorApplication.bindings.cs/]); (2) on 2021.2–2022.1 a held lock could silently skip the play-mode domain reload ([U2021–U2022 | HIGH | https://issuetracker.unity3d.com/issues/domain-reload-missing-when-entering-play-mode]); (3) without DisallowAutoRefresh, mid-turn imports can mutate assets under the AI's feet — adding the Disallow pair is the documented-counter-safe fix, cost is one extra pair + explicit `Refresh()` on release.
