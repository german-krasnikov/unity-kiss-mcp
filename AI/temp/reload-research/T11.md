# T11 — Per-OS Deltas for Unity Editor Reload/Recompile Automation (macOS / Windows / Linux)

## (a) Editor.log location per OS, -logFile, rotation

- [U2021.3–U6000.x | HIGH | https://docs.unity3d.com/Manual/log-files.html] Official Editor log paths: **Linux** `~/.config/unity3d/Editor.log`; **macOS** `~/Library/Logs/Unity/Editor.log`; **Windows** `%LOCALAPPDATA%\Unity\Editor\Editor.log`. Identical list in 2021.3 docs (https://docs.unity3d.com/2021.3/Documentation/Manual/LogFiles.html).
- [U2021.3–U6000.x | HIGH | https://docs.unity3d.com/Manual/log-files.html] Package Manager log (`upm.log`) sits next to Editor.log on macOS/Linux; on Windows it is `%LOCALAPPDATA%\Unity\Editor\upm.log` for user accounts but `%ALLUSERSPROFILE%\Unity\Editor\upm.log` when Unity runs under the SYSTEM account (relevant for CI services). Hub logs: `~/.config/UnityHub/logs/info-log.json` (Linux), `~/Library/Application Support/UnityHub/logs/info-log.json` (macOS), `%UserProfile%\AppData\Roaming\UnityHub\logs\info-log.json` (Windows).
- [U2021.3–U6000.x | HIGH | https://docs.unity3d.com/6000.0/Documentation/Manual/EditorCommandLineArguments.html] `-logFile <pathname>` redirects the Editor log to any path; `-logFile -` outputs to console; "On Windows you can use -logFile to direct the output to stdout, which by default is not the console." Automation implication: if the editor instance you tail was launched with `-logFile`, the default per-OS path above is NOT written — resolve the override first.
- [U2021.3–U6000.x | LOW | https://discussions.unity.com/t/my-editor-prev-log-is-over-8gb-can-i-safely-remove-it/99369] Rotation: on Editor start, the previous session's `Editor.log` is moved to `Editor-prev.log` in the same directory (one generation only, no size cap — community reports 8–40 GB files). NOT FOUND: any current docs.unity3d.com page documenting `Editor-prev.log` — the modern log-files page omits it; treat rotation as de-facto behavior, and handle the inode swap when tailing (reopen by path, don't hold the fd).
- NOT FOUND: documented per-OS differences in *content* written to Editor.log (Bee/compilation markers are platform-neutral; no official statement located either way).

## (b) Linux Editor specifics

- [U2021.3 | HIGH | https://docs.unity3d.com/2021.3/Documentation/Manual/system-requirements.html] 2021.3 Linux Editor officially supports Ubuntu 20.04, Ubuntu 18.04, CentOS 7; Gnome on **X11 only**; doc explicitly notes "File systems are case sensitive" and the `Pipe error !` → `ulimit -n 4096` workaround.
- [U6000.4 | HIGH | https://docs.unity3d.com/6000.4/Documentation/Manual/system-requirements.html] Unity 6.x supports Ubuntu 22.04/24.04; Gnome on X11 **or Wayland** (Wayland with driver caveats); same "File systems are case sensitive" note; video import limited to VP8.
- [U2020.1+ | MED | https://unity.com/releases/2020-1/editor-team-workflows] Directory Monitoring (fast non-scan Refresh) is documented as "available on Windows", with "support for macOS to be added in a future release" — **Linux is never mentioned**; inference (flagged): Linux has no inotify-based Directory Monitoring and refresh is full-scan, same as macOS. Corroborated by API docs listing only the `-DisableDirectoryMonitor` flag and symlink auto-disable, no Linux mention (https://docs.unity3d.com/2021.1/Documentation/ScriptReference/AssetDatabase.IsDirectoryMonitoringEnabled.html).
- [U6000.3.6–6000.3.8 | HIGH | https://issuetracker.unity3d.com/issues/linux-auto-refresh-fails-to-reimport-and-compile-script-changes-when-editing-files-outside-the-editor] Regression: Linux-only, Auto Refresh stops reimporting/compiling externally-edited scripts after a few cycles (Ubuntu 24.04/25.04, Mint 22.3; not reproducible on Windows 11). Fixed in 6000.3.10f1 / 6000.4.0b10 / 6000.5.0a8. Direct relevance: an external-MCP "edit file → focus editor → expect compile" loop silently dies on these Linux versions.
- [U2022.3.44–U6000.0.17 | HIGH | https://issuetracker.unity3d.com/issues/linux-editor-freezes-for-1-2-minutes-when-asset-database-is-refreshed] UUM-79033: Linux Editor freezes 1–2 min on AssetDatabase refresh when the open-file hard limit is very high (`ulimit -n -H 536870908`); regression, fixed 2022.3.55f1 / 6000.0.36f1. Combined with the 2021.3 docs telling you to *raise* ulimit, automation on Linux should pin a sane `ulimit -n` (e.g. 4096), not max it.
- [U6000.x | LOW | https://issuetracker.unity3d.com/issues/linux-having-same-case-insensitive-named-assets-causes-infinite-import-looping] "[Linux] Having same case-insensitive named assets causes infinite import looping" — assets differing only by case (legal on ext4) can loop the importer forever on Linux.

## (c) Windows-specific reload/socket pitfalls

- [MS Learn | HIGH | https://learn.microsoft.com/en-us/windows/win32/winsock/using-so-reuseaddr-and-so-exclusiveaddruse] Windows SO_REUSEADDR ≠ BSD semantics: on Windows it lets a second socket "forcibly bind to a port in use by another socket" with non-deterministic delivery (port hijack), whereas BSD uses it mainly for TIME_WAIT rebind. SO_EXCLUSIVEADDRUSE is the Windows-only hardening option; with it set, a closed listener's port "cannot bind … until the original connection becomes inactive" — i.e. exclusive-mode sockets are NOT immediately rebindable after close while connections drain. Practical delta: a rebind that works instantly on macOS/Linux can fail transiently on Windows right after domain reload.
- [.NET API | HIGH | https://learn.microsoft.com/en-us/dotnet/api/system.net.sockets.socket.exclusiveaddressuse] `Socket.ExclusiveAddressUse` "default is `true` for Windows Server 2003 and Windows XP and newer"; must be set before `Bind`. The page documents Windows behavior only; NOT FOUND: official documentation of this property's semantics on Unix under Mono (Unity's editor runtime) — treat cross-platform behavior of this flag as unspecified and instead retry-bind with backoff.
- [CoplayDev/unity-mcp | LOW | https://github.com/CoplayDev/unity-mcp/issues/1173] Field evidence from a competing Unity-MCP bridge: TcpListener leak after domain reload on **Windows specifically**; a 500 ms wait for the old listener task to release the socket proved too short (was 2000 ms), producing AddressAlreadyInUse and a silent port-fallback. Community workaround in related threads: disable Reload Domain (no reload → no socket churn) (https://discussions.unity.com/t/socketexception-address-already-in-use-system/898620, LOW). Budget ≥2 s release-wait or bind-retry loop on Windows.
- [U2021–U6000 | HIGH | https://docs.unity3d.com/Packages/com.unity.asset-store-validation@0.1/manual/path-length-validation.html] Official Unity package-validation doc: "Unity's subsystems does not work with long path on Windows"; package file paths capped at 140 chars precisely because `Library/PackageCache` expansion can blow MAX_PATH (260, per https://learn.microsoft.com/en-us/windows/win32/fileio/maximum-file-path-limitation). Issue-tracked example: visual-scripting paths in `Library\PackageCache` exceeding the OS limit (https://issuetracker.unity3d.com/issues/long-visualscripting-path-can-result-in-exceeding-os-character-limit). Automation rule: keep Windows project roots short (e.g. `C:\U\proj`), especially for the headless project copy.
- [MS Learn | HIGH | https://learn.microsoft.com/en-us/defender-endpoint/microsoft-defender-endpoint-antivirus-performance-mode] Defender: Microsoft's documented mitigation is Dev Drive (ReFS) + Defender "performance mode" (async scanning, enabled by default on trusted Dev Drives; Microsoft cites up to 30% build-time improvement — https://devblogs.microsoft.com/visualstudio/devdrive/). Defender has also false-positive-flagged files in `Library/Bee` artifacts (community, LOW: https://discussions.unity.com/t/file-in-library-bee-folder-flagged-as-virus-by-defender/1688645). NOT FOUND: an official docs.unity3d.com page recommending antivirus exclusions for import/compile speed.

## (d) Filesystem semantics deltas

- [Apple | HIGH | https://support.apple.com/guide/disk-utility/file-system-formats-dsku19ed921c/mac] macOS default APFS is case-insensitive; "APFS (Case-sensitive)" is a separate opt-in variant. So default macOS == Windows (insensitive), Linux == sensitive — per Unity's own Linux sysreq note "File systems are case sensitive" (https://docs.unity3d.com/6000.4/Documentation/Manual/system-requirements.html).
- [U2020.3–U2021.2 | HIGH | https://issuetracker.unity3d.com/issues/moving-file-failed-error-when-renaming-an-asset-and-only-changing-the-case-of-the-characters-in-windows-7] Case-only renames of assets produced "Moving file failed" errors on Windows (regression window 2020.3.7–2021.2.0a17). Combined with the Linux infinite-import-loop bug above: never generate files differing only by case, and never do case-only renames via raw file I/O — refresh behavior diverges per OS.
- [U2021–U6000 | LOW | https://discussions.unity.com/t/unity-meta-file-inconsistent-casing-renaming-script/809214] Renaming a .cs in the IDE with case changes yields "inconsistent casing" warnings and Unity reverting to the old name on case-insensitive FS (macOS/Windows) — meta/asset casing mismatch breaks the script-class association that recompile relies on.
- [U2021–U6000 | LOW | https://discussions.unity.com/t/unity-2020-2-compatible-with-symlink/823112] Symlinks in projects: Unity's stance is unsupported/at-your-own-risk (editor shows a corruption warning; support article uses symlinks only as an exclusion hack with "THIS IS NOT SUPPORTED. USE AT YOUR OWN RISK": https://support.unity.com/hc/en-us/articles/208456906-Excluding-Scripts-and-Assets-from-builds). Hard documented consequence: "Directory Monitoring is automatically disabled when Unity detects symlinks in the project" (HIGH, https://docs.unity3d.com/2021.1/Documentation/ScriptReference/AssetDatabase.IsDirectoryMonitoringEnabled.html) — a single symlink silently degrades Windows refresh to full-scan mode.
- [U2021–U6000 | HIGH | https://docs.unity3d.com/Manual/upm-localpath.html] `file:` package paths: absolute or relative-to-`Packages/`; always forward slashes; Windows absolute form is `file:C:/...` (drive letter after prefix) — so cross-drive `file:` packages are officially expressible on Windows via absolute paths. NOT FOUND: official stance on UNC/network paths or symlinked `file:` targets on that page.
- NOT FOUND: any official Unity documentation tying file-mtime timestamp *granularity* (NTFS 100ns vs ext4 ns vs FAT 2s, etc.) to asset-change detection. Do not build logic on sub-second mtime deltas.

## (e) Per-OS process/launch facts for automation

- [U2021.3–U6000.x | HIGH | https://docs.unity3d.com/6000.0/Documentation/Manual/EditorCommandLineArguments.html] Official Hub-managed editor binaries: **Windows** `C:\Program Files\Unity\Hub\Editor\<Version>\Editor\Unity.exe`; **macOS** `/Applications/Unity/Hub/Editor/<Version>/Unity.app/Contents/MacOS/Unity` (note: the binary inside the .app, not `open -a`); **Linux** `/opt/Unity/Hub/Editor/<Version>/Editor/Unity` per docs — but community reports the Linux Hub actually defaults to `~/Unity/Hub/Editor` and the path is user-configurable (LOW, https://discussions.unity.com/t/unity-hub-for-linux-doesnt-allow-custom-install-location/739406); resolve via Hub CLI/preferences rather than hardcoding on Linux.
- [U2021.3–U6000.x | HIGH | https://docs.unity3d.com/6000.0/Documentation/Manual/EditorCommandLineArguments.html] `-projectPath` accepts absolute or relative paths on all OS (quote if spaces); `-quit` + `-batchmode` is the documented graceful exit; `-quitTimeout` (default 300 s) bounds waiting on async tasks; exit code 1 on script exceptions in batch mode; `EditorApplication.Exit(code)` for custom codes. Several `-force-*` graphics flags are OS-gated (d3d11/d3d12/gles Windows-only, metal macOS-only).
- [U2021–U6000 | MED | https://issuetracker.unity3d.com/issues/unity-server-run-time-does-not-respond-to-sigterm-and-other-linux-signals-correctly] Unity (server runtime/player) is issue-tracked as not responding correctly to SIGTERM and other Linux signals; community workaround is a native signal-handler plugin calling Application.Quit (LOW, https://sdumetz.github.io/2017/07/01/handle-unix-signals-unity.html). NOT FOUND: any official documentation of EDITOR SIGTERM/SIGINT handling on any OS — for graceful headless shutdown, rely on `-quit`/`EditorApplication.Exit`, not signals; reserve SIGKILL/taskkill as last resort (expect Temp lockfile + "project already open" cleanup: https://support.unity.com/hc/en-us/articles/40828087523092-Resolving-the-The-project-is-currently-open-in-the-Unity-Editor-Please-close-it-in-the-Editor-to-proceed-with-this-operation-Error).
- [U2021–U6000 | LOW | https://support.unity.com/hc/en-us/articles/115003118426-Running-multiple-instances-of-Unity-referencing-the-same-project] One project folder can be opened by only one editor instance on every OS; Unity's support article for "second instance" workflows is a symlinked project copy (Assets/ProjectSettings/Packages shared, separate Library) — unsupported-tier guidance, but the only documented multi-instance pattern for a headless compile copy.
