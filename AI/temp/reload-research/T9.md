# T9 — "Compile finished" signals & event ordering (Unity 2021.3 → 6000.x)

## (a) Event ordering and which side of the reload

- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/ScriptReference/Compilation.CompilationPipeline-compilationStarted.html] `compilationStarted` is invoked on the main thread when compilation of assemblies starts, and is called before the first `assemblyCompilationStarted` event.
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.3/Documentation/ScriptReference/Compilation.CompilationPipeline-compilationFinished.html] `compilationFinished` is "invoked on the main thread when the compilation of assemblies finishes" and fires "after the last CompilationPipeline.assemblyCompilationFinished event"; triggering Player builds or additional compilation from these callbacks is unsupported.
- [U2021.3–U6000.4 | HIGH | https://docs.unity3d.com/ScriptReference/AssemblyReloadEvents-beforeAssemblyReload.html] `beforeAssemblyReload` "is dispatched just before Unity reloads all assemblies"; reload happens "when scripts have changed and when entering play mode" — i.e. it runs in the OLD domain.
- [U2021.3–U6000.4 | HIGH | https://docs.unity3d.com/ScriptReference/AssemblyReloadEvents-afterAssemblyReload.html] `afterAssemblyReload` is dispatched "just after Unity have reloaded all assemblies" — i.e. it runs in the NEW domain.
- [U2021.3–U6000.3 | MED | https://hextantstudios.com/unity-log-compile-times/] Empirical ordering used in tooling: `compilationStarted/Finished` fire pre-reload (their elapsed time must be persisted across the reload to be reported), then `beforeAssemblyReload` → [reload] → `afterAssemblyReload`; their code even guards "Return if the assembly was reloaded before timers were started".
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.3/Documentation/ScriptReference/Callbacks.DidReloadScripts-ctor.html] `[DidReloadScripts]` has a `callbackOrder` parameter; lower runs earlier, and Unity's internal processing sits at order 0 — it runs in the NEW domain after reload.
- [U2021.3–U6000.x | LOW | https://discussions.unity.com/t/initializeonload-vs-didreloadscripts/572660] Community testing: "`InitializeOnLoad` and `InitializeOnLoadMethod` are always called before `DidReloadScripts`", but relative order among multiple `InitializeOnLoad`s is undefined.
- [U2021.3–U6000.x | LOW | https://discussions.unity.com/t/callback-after-assembly-reloads-but-before-scriptable-objects-on-enable/919319] Community: `afterAssemblyReload` fires after `OnEnable` of reload-reconstructed objects.
- NOT FOUND: an official single page guaranteeing the full chain `compilationStarted → … → compilationFinished → beforeAssemblyReload → reload → afterAssemblyReload` as a contract; a user asking exactly this got no official answer — [U2021.3–U6000.x | LOW | https://discussions.unity.com/t/whats-the-exact-order-of-script-compilation/920018] "I want to know the exact execute order of a script compilation, however, it's not documented."

## (b) Failure path — THE critical truth

- [U2021.3–U6000.4 | HIGH | https://docs.unity3d.com/ScriptReference/Compilation.CompilationPipeline.RequestScriptCompilation.html] Canonical docs statement: "After the compilation, **if the compilation was successful**, the Editor reloads all assemblies" — domain reload is CONDITIONAL on success; on compile errors NO reload happens and the process keeps running the OLD assemblies.
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.3/Documentation/ScriptReference/Compilation.CompilationPipeline-assemblyCompilationFinished.html] `assemblyCompilationFinished` fires per assembly even on failure: "Assembly build failed if there any build errors"; check the `CompilerMessage[]` for errors.
- [U2021.3–U6000.x | MED | https://github.com/Unity-Technologies/UnityCsReference/blob/master/Editor/Mono/Scripting/ScriptCompilation/CompilationPipeline.cs] In UnityCsReference, `compilationFinished`/`assemblyCompilationFinished` are forwarded unconditionally from `EditorCompilation` (wrapped only in try/catch logging) — so `compilationFinished` fires for FAILED compiles too; a handler that flips "compile done" on `compilationFinished` will run while the editor still executes OLD code.
- [U2018–U2021 | MED | https://discussions.unity.com/t/custom-assemblies-are-not-reloaded-if-there-is-a-compile-error/705717] Observed: "'Begin MonoManager ReloadAssembly' does not get raised after compilation is completed with errors" — even successfully-compiled asmdef assemblies are not loaded until ALL errors are fixed (despite docs wording suggesting otherwise).
- [U2018–U6000.x | LOW | https://discussions.unity.com/t/initializeonload-and-didreloadscripts-not-running-when-net4-6-errors-prevent-compile/707907] `[DidReloadScripts]` "only runs if there are no compile errors"; `[InitializeOnLoad]` re-runs only on editor open in that broken state — no reload-side callback fires on a failed compile.
- [U2021.3–U6000.4 | HIGH | https://docs.unity3d.com/ScriptReference/EditorUtility-scriptCompilationFailed.html] `EditorUtility.scriptCompilationFailed`: "True if there are any compilation error messages in the log" — the cheap post-compile success/failure check.
- [U2021.3–U6000.x | MED | https://issuetracker.unity3d.com/issues/editorutility-dot-scriptcompilationfailed-not-flagging-package-compilation-errors-during-editor-startup] Known bug: `scriptCompilationFailed` did not flag PACKAGE compile errors during editor startup — don't trust it as the sole signal at boot.
- [U2021.3–U6000.4 | HIGH | https://docs.unity3d.com/6000.4/Documentation/Manual/SafeMode.html] On OPENING a project with compile errors Unity offers Safe Mode: no managed project/package code runs at all, and Unity auto-exits Safe Mode when errors are resolved (full reimport then).

## (c) External observability & polling pitfalls

- [U2021.3–U6000.4 | HIGH | https://docs.unity3d.com/ScriptReference/EditorApplication-isCompiling.html] `isCompiling` is documented ONLY as "Is editor currently compiling scripts? (Read Only)" — it makes no statement about the domain-reload phase, so `!isCompiling` ≠ "new code is live"; combined with the conditional-reload fact above, polling it has a false-"done" window after a failed compile and between compile end and reload.
- [U2019–U2021 | LOW | https://discussions.unity.com/t/editorapplication-iscompiling-always-true-with-recompile-after-finished-playing-preference/739571] Known wobble: with "Recompile After Finished Playing", `isCompiling` got stuck true forever, breaking plugins that poll it.
- [U2019.3 | LOW | https://discussions.unity.com/t/editorapplication-iscompiling-is-always-false/763929] Inverse wobble: in 2019.3 beta `isCompiling` read always-false from some contexts — another reason not to build a handshake on it alone.
- [U2021.3–U6000.x | LOW | https://feedback.vrchat.com/sdk-bug-reports/p/vrchat-package-resolver-tool-uses-hacky-workarounds-to-initialize] Real-world pattern (called "hacky" even by its users): wait until both `EditorApplication.isCompiling` and `EditorApplication.isUpdating` are false to consider the editor settled.
- NOT FOUND: any official, documented signal observable from OUTSIDE the editor process that the domain reload completed. Community substitutes:
- [U2021.3–U6000.x | MED | https://github.com/IvanMurzak/Unity-MCP/wiki/Troubleshooting] Bridge pattern: "Unity-MCP disconnects before a domain reload and reconnects afterward… Brief disconnections during these transitions are normal" — the TCP drop (in `beforeAssemblyReload`) + reconnect (from the new domain) IS the external reload-complete signal.
- [U2021.3–U6000.x | MED | https://github.com/CoplayDev/unity-mcp/issues/1173] Same architecture in CoplayDev/unity-mcp: TCP listener is torn down and restarted across domain reload — external clients detect reload completion by the listener coming back.
- [U2021.3–U6000.x | MED | https://github.com/CoplayDev/unity-mcp/discussions/241] Maintainer statement: "Compiling isn't enough, unity actually needs to reload its domain so the assembly from the compiled files are activated" — their `manage-script` flow explicitly waits for compile AND reload.
- [U2021.3–U6000.x | MED | https://issuetracker.unity3d.com/issues/editor-freezes-at-begin-monomanager-reloadassembly-when-starting-a-scene-after-a-few-times] `Begin MonoManager ReloadAssembly` is a real Editor.log line at reload start — tailing Editor.log gives an out-of-band reload marker (paired with the subsequent `Domain Reload Profiling` block, see https://discussions.unity.com/t/editor-stuck-on-domain-reload-or-complete-domain-burst-threads/1692576).
- [U2021.3–U6000.4 | MED | https://docs.unity3d.com/ScriptReference/EditorUtility.RequestScriptReload.html] `EditorUtility.RequestScriptReload` exists to force a domain reload explicitly when you need a guaranteed reload pass.

## (d) Which callback carries the errors; deprecations

- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.3/Documentation/ScriptReference/Compilation.CompilationPipeline-assemblyCompilationFinished.html] `assemblyCompilationFinished` = `Action<string, CompilerMessage[]>` (assembly path + compiler messages) — this is the ONLY per-compile event that carries errors.
- [U2021.3–U6000.4 | HIGH | https://docs.unity3d.com/ScriptReference/Compilation.CompilerMessage.html] `CompilerMessage` fields: `message`, `file`, `line`, `column`, `type` (`CompilerMessageType` Error/Warning).
- [U2021.3–U6000.3 | HIGH | https://docs.unity3d.com/6000.3/Documentation/ScriptReference/Compilation.CompilationPipeline-compilationFinished.html] `compilationFinished`'s parameter is an opaque `object` context: "a token generated by Unity, which uniquely identifies a single compilation cycle", usable only to pair with `compilationStarted` — it carries NO error info; aggregate errors from `assemblyCompilationFinished` or read `EditorUtility.scriptCompilationFailed`.
- [U2021.1–U6000.x | HIGH | https://github.com/Unity-Technologies/UnityCsReference/blob/master/Editor/Mono/Scripting/ScriptCompilation/CompilationPipeline.cs] `assemblyCompilationStarted` is `[Obsolete("Use compilationStarted, compilationFinished or assemblyCompilationFinished instead. Note that using any of these functions to do time measurements is a bad idea as they run async to actual compilation.")]`; `assemblyCompilationFinished` is NOT obsolete; `compilationStarted/Finished` are `Action<object>`.

## (e) Handshake patterns for "new code is live"

- [U2021.3–U6000.4 | HIGH | https://docs.unity3d.com/ScriptReference/SessionState.html] `SessionState` is "a Key-Value Store intended for storing and retrieving Editor session state that should survive assembly reloading" (cleared on editor exit) — the documented primitive for a reload-crossing token (write pre-reload, read in `afterAssemblyReload`/`[InitializeOnLoad]`).
- [U2021.3–U6000.3 | MED | https://hextantstudios.com/unity-log-compile-times/] Working sample: measure `compilationStarted→compilationFinished` (compile) and `beforeAssemblyReload→afterAssemblyReload` (reload) with state persisted across the reload via a serialized editor singleton — proves the two-phase handshake structure.
- [U2021.3–U6000.x | MED | https://discussions.unity.com/t/how-to-tell-that-compilation-exceptions-are-resolved-when-no-recompiling-occurred/1576369] Trap: reverting a file to its last-compiled content can SKIP recompilation entirely (content-hash check) — no `assemblyCompilationFinished` fires; `AssemblyReloadEvents.afterAssemblyReload` was suggested as the more reliable "code is current" hook.
- [U2021.3–U6000.x | LOW | https://discussions.unity.com/t/apply-shader-after-editor-compiled/859164] Community guidance: prefer the `compilationFinished` callback over coroutine/`delayCall` polling of `isCompiling` for in-editor post-compile work.

## Definitive state machine (each transition with its observable)

```
EDIT ──AssetDatabase.Refresh/RequestScriptCompilation──▶ COMPILING
  signal: compilationStarted (old domain)                [HIGH, docs]
COMPILING ─▶ COMPILED-OK or COMPILED-ERROR
  signal: N× assemblyCompilationFinished(path, CompilerMessage[]) then
          compilationFinished(ctx) — fires in BOTH outcomes [HIGH docs + MED CsRef]
  discriminator: EditorUtility.scriptCompilationFailed / any CompilerMessageType.Error [HIGH, docs]
COMPILED-ERROR ─▶ (terminal) NO domain reload; OLD assemblies keep running;
  no beforeAssemblyReload/afterAssemblyReload/DidReloadScripts fire   [HIGH docs ("if successful, reloads") + MED forum]
COMPILED-OK ─▶ RELOADING
  signal: beforeAssemblyReload (last code in old domain; close sockets here) [HIGH, docs]
RELOADING ─▶ RELOADED (new code live)
  signal in-process: InitializeOnLoad → DidReloadScripts → afterAssemblyReload-region (new domain) [HIGH docs + LOW order]
  signal external: TCP listener drop+reconnect, or Editor.log "Begin MonoManager ReloadAssembly"/"Domain Reload Profiling" [MED]
```

Bottom line: "compile done" (`compilationFinished`) and "new code running" (`afterAssemblyReload` in the new domain) are DIFFERENT states separated by a conditional reload that is skipped entirely on error; only the success path ever reaches the reload events, and no official externally-visible completion signal exists — bridge reconnect or Editor.log markers are the de-facto substitutes.
