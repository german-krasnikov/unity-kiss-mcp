# T10 — Pitfall Catalog: Editor Reload Automation (Unity 2021.3 LTS → 6000.x, macOS host)

## (a) Stale `Library/ScriptAssemblies` — Unity runs old code after "successful" refresh

- [U2021.3–U6000 | HIGH | https://docs.unity3d.com/6000.1/Documentation/Manual/AssetDatabaseRefreshing.html] SYMPTOM: code after a scripted `AssetDatabase.Refresh()` still executes OLD assemblies | CAUSE: refresh step 3 is verbatim "Reload the scripting domain, if Refresh was not invoked from a script" — a script-invoked Refresh compiles but does NOT reload the domain inline; reload happens later (caller's frames are in the old domain) | DETECTION: `AssemblyReloadEvents.afterAssemblyReload` never fired since your Refresh | MITIGATION: treat scripted Refresh as async — return from the TCP handler, wait for reload event before reporting "done".
- [U2021.3–U6000 | HIGH | https://docs.unity3d.com/6000.1/Documentation/Manual/AssetDatabaseRefreshing.html] SYMPTOM: file written by external tool, Unity never compiles | CAUSE: refresh only runs "When the Unity Editor regains focus (if you have enabled Auto-Refresh in the Preferences window)", via menu, or via scripted Refresh — an unfocused headless/background editor (typical macOS automation) never auto-refreshes | DETECTION: `EditorApplication.isCompiling` stays false after file write | MITIGATION: always call `AssetDatabase.Refresh()` from the plugin instead of relying on focus.
- [U2021.3–U6000 | MED | https://discussions.unity.com/t/custom-assemblies-are-not-reloaded-if-there-is-a-compile-error/705717] SYMPTOM: console shows errors in ONE asmdef, yet edits to OTHER (clean) assemblies also don't take effect | CAUSE: when any assembly fails to compile, no assembly reload occurs ("Begin MonoManager ReloadAssembly" never raised), contrary to per-assembly expectations | DETECTION: compile-error scan + missing reload event | MITIGATION: gate all "code updated" reporting on zero compile errors, not on per-file compile success.
- [U2021.3–U6000 | MED | https://issuetracker.unity3d.com/issues/assemblies-not-being-reloaded-when-reimporting-c-number-script-asset] SYMPTOM: reimporting a .cs asset doesn't produce a fresh assembly | CAUSE: known issue — assemblies not reloaded on script reimport | DETECTION: compare behavior vs expected new code path | MITIGATION: force via `CompilationPipeline.RequestScriptCompilation()` (MED, unfetched stable URL: https://docs.unity3d.com/ScriptReference/Compilation.CompilationPipeline.RequestScriptCompilation.html).
- [U2021.3–U6000 | MED | https://discussions.unity.com/t/stuck-on-reloading-script-assemblies/907803] SYMPTOM: editor hangs forever on "Reloading Script Assemblies" after compile | CAUSE: failed/deadlocked reload (community reports; only fix reported is force-killing the editor) | DETECTION: out-of-process watchdog timeout on TCP heartbeat | MITIGATION: external watchdog + restart; never wait unbounded for reload.
- [U2021.3–U6000 | MED | https://docs.unity3d.com/6000.1/Documentation/Manual/SafeMode.html] SYMPTOM: weird stale artifacts after a Safe Mode session was force-exited | CAUSE: docs warn ignoring errors risks "incorrect cached artifacts in your Library or in your Cache Server" | DETECTION: behavior mismatch after a Safe Mode exit-with-errors | MITIGATION: reimport/clear Library after any forced Safe Mode exit.
- [U2021.3–U6000 | MED | https://docs.unity3d.com/ScriptReference/AssemblyReloadEvents.html] DETECTION TOOLBOX: `AssemblyReloadEvents.beforeAssemblyReload/afterAssemblyReload`, `EditorApplication.isCompiling` (https://docs.unity3d.com/ScriptReference/EditorApplication-isCompiling.html), `CompilationPipeline` compilation events (https://docs.unity3d.com/ScriptReference/Compilation.CompilationPipeline.html) are the official hooks to prove "new code is actually loaded".
- NOT FOUND: any documented Unity API or official recipe for comparing "dll on disk in `Library/ScriptAssemblies` is newer/older than the loaded assembly" (timestamp vs `Assembly.Location`); community practice exists but no citable authoritative source.

## (b) Safe Mode

- [U2021.3–U6000 | HIGH | https://docs.unity3d.com/6000.1/Documentation/Manual/SafeMode.html] ENTRY: Unity offers/enters Safe Mode when opening a project that has script compilation errors (version upgrade, missing packages, broken scripts, VCS pull).
- [U2021.3–U6000 | HIGH | https://docs.unity3d.com/6000.1/Documentation/Manual/SafeMode.html] SYMPTOM: TCP plugin silently absent after editor restart | CAUSE: "Safe Mode never allows managed code to run from your project, or its packages" and "your own scripts such as Editor scripts, asset post-processors, and scripted importers do not run" — your in-package TCP server NEVER loads in Safe Mode; only Console/Project/Inspector/Package Manager windows work | DETECTION: out-of-process only (port 9500 never opens) | MITIGATION: external watchdog interprets "editor alive + port closed for N min" as probable Safe Mode/compile failure.
- [U2021.3–U6000 | HIGH | https://github.com/Unity-Technologies/UnityCsReference/blob/master/Editor/Mono/EditorUtility.bindings.cs] DETECTION API: `EditorUtility.isInSafeMode` exists but is `internal static extern bool isInSafeMode` (FreeFunction `GetApplication().IsInSafeMode`) — public docs page 404s; reachable only via reflection, and moot anyway since your code doesn't run in Safe Mode.
- [U2021.3–U6000 | HIGH | https://docs.unity3d.com/6000.1/Documentation/Manual/SafeMode.html] EXIT: automatic when no compile errors remain; manual via toolbar "Exit Safe Mode" button (exiting with errors remaining is "not recommended").
- [U2021.3–U6000 | HIGH | https://docs.unity3d.com/6000.1/Documentation/Manual/SafeMode.html] BATCH MODE: "In Batch Mode, Unity automatically quits if there are compilation errors in your project, unless you use the `-ignoreCompilerErrors` command line argument" — headless automation gets a dead process, not Safe Mode.
- NOT FOUND: any public API or command-line switch to exit Safe Mode programmatically (no `ExitSafeMode` in UnityCsReference search results; only the GUI button).

## (c) Refresh/recompile during Play Mode

- [U2021.3–U6000 | HIGH | https://docs.unity3d.com/6000.1/Documentation/Manual/Preferences.html] The three "Script Changes While Playing" values: "Recompile And Continue Playing" ("default behavior… you might want to change it if your scripts rely on any non-serializable data" — domain reloads mid-play, non-serialized state lost), "Recompile After Finished Playing" ("Defer recompilation until you manually stop your scene"), "Stop Playing And Recompile" ("Immediately stops your scene for recompilation").
- [U2021.3–U2021.3.24 | HIGH | https://issuetracker.unity3d.com/issues/script-recompiles-in-play-mode-when-script-changes-while-playing-option-is-set-to-recompile-after-finished-playing-and-auto-refresh-is-set-to-enabled] SYMPTOM: scripts recompile mid-play despite "Recompile After Finished Playing" | CAUSE: bug UUM-20409 when Auto Refresh=Enabled (repro 2021.1.0a8–2023.1.0a20; FIXED 2021.3.25f1/2022.2.x/2023.x) | DETECTION: unexpected domain reload event during play | MITIGATION: on affected versions set Auto Refresh to "Enabled Outside Play Mode" ("seems to stop the recompilation in Play Mode" per the report).
- [U2021.3–U6000 | MED | https://docs.unity3d.com/6000.1/Documentation/Manual/Preferences.html] A scripted `AssetDatabase.Refresh()` mid-play still imports changed assets; whether resulting script compilation applies immediately, defers, or stops play is governed by the ScriptCompilationDuringPlay preference above — under "Recompile And Continue Playing" expect a mid-play domain reload triggered by your own automation call.
- [U2021.3–U6000 | MED | https://www.jetbrains.com/help/rider/Refreshing_Unity_Assets.html] BEST PRACTICE precedent: "Rider does not refresh assets if Unity is in the play mode" — professional tooling deliberately queues refresh until play exits; automation should do the same instead of trusting the preference.

## (d) AssetDatabase from non-main thread

- [U2021.3–U6000 | HIGH | https://docs.unity3d.com/6000.2/Documentation/Manual/job-system-overview.html] REQUIREMENT: most Unity APIs aren't thread-safe and can only be called from the main thread; Unity raises an error if called from other threads — applies to all of `AssetDatabase`.
- [U2021.3–U6000 | LOW | https://forum.unity.com/threads/assetdatabase-createasset-and-multithreading.446610/] FAILURE MODE: calling AssetDatabase APIs from a worker thread throws a managed exception ("…can only be called from the main thread"), it does not silently corrupt — but a TCP-socket thread that swallows exceptions LOOKS like silent failure | DETECTION: wrap handlers, log exceptions | MITIGATION: marshal first, call second.
- [U2021.3–U6000 | HIGH | https://docs.unity3d.com/6000.0/Documentation/Manual/async-awaitable-continuations.html] MARSHALING 1: "Unity overwrites the default SynchronizationContext with a custom UnitySynchronizationContext to ensure all .NET Task continuations in both Edit mode and Play mode run on the main thread by default" — capture the main-thread `SynchronizationContext` at load and `Post()` work to it from the socket thread.
- [U6000–U6000 | HIGH | https://docs.unity3d.com/6000.0/Documentation/Manual/async-awaitable-continuations.html] MARSHALING 2 (Unity 6 canonical): `await Awaitable.MainThreadAsync()` guarantees "continuation happens on the main thread"; `Awaitable.BackgroundThreadAsync()` hops back off.
- [U2021.3–U6000 | MED | https://docs.unity3d.com/ScriptReference/EditorApplication-delayCall.html] MARSHALING 3: classic patterns — enqueue a closure into a thread-safe queue drained by `EditorApplication.update` (https://docs.unity3d.com/ScriptReference/EditorApplication-update.html), or `EditorApplication.delayCall` for one-shot main-thread execution; works in all target versions (URLs stable, pages unfetched).

## (e) Recursive import loops & Burst × domain reload

- [U2021.3–U6000 | HIGH | https://docs.unity3d.com/6000.1/Documentation/ScriptReference/AssetPostprocessor.OnPostprocessAllAssets.html] RE-ENTRANCY: "If your code causes any new asset imports during this callback, OnPostProcessAllAssets will be called again once those new imports are complete" — writing/importing assets in the callback recurses by design.
- [U2021.3–U6000 | HIGH | https://docs.unity3d.com/6000.1/Documentation/Manual/AssetDatabaseRefreshing.html] LOOP FUEL: Unity restarts the whole refresh if "After the import, a file that the importer used has changed on disk", "The last modified timestamp of the file being imported changes while it's being imported", or "An importer creates a file in the middle of an import" — any tool writing into Assets/ during import restarts refresh indefinitely.
- [U2021.3–U6000 | MED | https://docs.unity.cn/Manual/AssetDatabaseRefreshing.html] PREVENTION (docs): forcing reimport inside pre/post-process callbacks via `ForceReserializeAssets`/`SaveAndReimport` requires care "not to cause infinite reimport loops" — only write when content actually differs, or write outside Assets/ and copy in one guarded pass.
- [U2021.3–U6000 | MED | https://issuetracker.unity3d.com/issues/an-infinite-import-loop-has-been-detected-when-processing-post-process-all-assets-callbacks-is-thrown-when-importing-stingray-pbs-from-maya] DETECTION: Unity has a built-in breaker that logs "An infinite import loop has been detected when processing post process all assets callbacks" — automation should grep console/Editor.log for this exact marker.
- [U2021.3–U6000 | MED | https://issuetracker.unity3d.com/issues/a-polybrushmesh-with-an-assetpostprocessor-causes-an-infinite-import-loop-whes-exclusively-checked-out-in-perforce] EDGE CASE: read-only files (Perforce exclusive checkout) make the postprocessor's write fail each cycle → infinite loop; check writability before writing in import callbacks.
- [U2021.3–U6000 | HIGH | https://docs.unity3d.com/Packages/com.unity.burst@1.8/changelog/CHANGELOG.html] BURST COST: Burst 1.8.1 changelog: "Changed how we handle domain reloads within Burst to avoid paying a 250ms cost on each domain reload"; 1.8.4 fixed "Domain completed stall when switching between debug/release scripting modes"; 1.8.0-pre.1 fixed a synchronous-compilation playmode-cycle hang — pin Burst ≥1.8.4 to avoid known reload stalls.
- [U2021.3–U6000 | MED | https://blog.s-schoener.com/2023-08-16-why-your-unity-project-is-slow/] BURST BEHAVIOR: "Burst compilation happens after regular compilation and usually happens in the background. It could be promoted to a blocking foreground process when you enter playmode or when you use the options for synchronous compilation (so don't use them!)" — automation entering play right after a refresh can hit a long "Burst is compiling" foreground stall.
- [U2021.3–U6000 | LOW | https://discussions.unity.com/t/editor-stuck-on-domain-reload-or-complete-domain-burst-threads/1692576] SYMPTOM: editor appears stuck on "Domain Reload"/"Complete Domain" with Burst threads active (community reports incl. Unity 6) | MITIGATION: wait-with-timeout, and for CI/automation disable Burst via `UNITY_BURST_DISABLE_COMPILATION` env var (referenced in Burst 1.8.22 changelog, same changelog URL as above).
